### Блок 1: Углубление в MCP (Model Context Protocol) (25 минут)

#### 1.1 Архитектура MCP серверов (10 минут)

**Что такое MCP сервер:**
- MCP сервер — это стандартизированный способ предоставления инструментов (tools) для LLM
- Работает по протоколу JSON-RPC 2.0
- Поддерживает два типа ресурсов: tools и resources

**Компоненты MCP сервера:**

1. **Tools (Инструменты)**
   - Функции, которые агент может вызывать
   - Описываются через JSON Schema
   - Имеют имя, описание и параметры
   - Пример: `get_sales`, `get_metadata`, `search_documents`

2. **Resources (Ресурсы)**
   - Данные, к которым агент может получить доступ
   - Могут быть файлами, URL, метаданными
   - Пример: документация, схемы БД, конфигурации

3. **Prompts (Промпты)**
   - Готовые шаблоны промптов для LLM
   - Могут включать переменные
   - Упрощают работу с агентами

**Протокол взаимодействия:**

```
Клиент (n8n, Continue, Cursor) 
    ↓
[Инициализация] → MCP Server
    ↓
[Список доступных tools] ← MCP Server
    ↓
[Вызов tool с параметрами] → MCP Server
    ↓
[Результат выполнения] ← MCP Server
```

**Преимущества MCP:**
- Единый стандарт для всех клиентов
- Не нужно писать интеграцию для каждого клиента отдельно
- Легко тестировать и отлаживать
- Можно комбинировать несколько MCP серверов

#### 1.2 MCP сервер для 1С - детали реализации (10 минут)

**Архитектура расширения 1С для MCP:**

```
HTTP-сервис mcp_APIBackend
    ↓
Обработка запросов MCP протокола
    ↓
Подсистема mcp_КонтейнерыИнструментов
    ↓
Обработки-контейнеры инструментов
    ├── Обработка1 (get_sales, get_metadata)
    ├── Обработка2 (search_documents)
    └── Обработка3 (пользовательские инструменты)
```

**Ключевые методы обработки-контейнера:**

1. **ДобавитьИнструменты(Инструменты)**
   - Описывает доступные инструменты
   - Возвращает массив структур с описанием
   - Вызывается при инициализации MCP клиента

2. **ВыполнитьИнструмент(ИмяИнструмента, Аргументы)**
   - Выполняет конкретный инструмент
   - Получает параметры из аргументов
   - Возвращает результат в формате MCP

**Пример структуры инструмента:**

```bsl
Процедура ДобавитьИнструменты(Инструменты) Экспорт
    
    ОписаниеИнструмента = Новый Структура;
    ОписаниеИнструмента.Вставить("name", "get_sales");
    ОписаниеИнструмента.Вставить("description", 
        "Получение данных о продажах из регистра бухгалтерии за указанный период");
    
    Параметры = Новый Массив;
    
    ПараметрНачало = Новый Структура;
    ПараметрНачало.Вставить("name", "startDate");
    ПараметрНачало.Вставить("type", "string");
    ПараметрНачало.Вставить("description", "Дата начала периода в формате ISO 8601 (YYYY-MM-DD)");
    Параметры.Добавить(ПараметрНачало);
    
    ПараметрКонец = Новый Структура;
    ПараметрКонец.Вставить("name", "endDate");
    ПараметрКонец.Вставить("type", "string");
    ПараметрКонец.Вставить("description", "Дата окончания периода в формате ISO 8601 (YYYY-MM-DD)");
    Параметры.Добавить(ПараметрКонец);
    
    ОписаниеИнструмента.Вставить("inputSchema", Новый Структура(
        "type", "object",
        "properties", Параметры,
        "required", Новый Массив("startDate")
    ));
    
    Инструменты.Добавить(ОписаниеИнструмента);
    
КонецПроцедуры
```

**Важные моменты:**
- Инструменты должны быть идемпотентными (повторный вызов дает тот же результат)
- Обработка ошибок обязательна
- Валидация параметров перед выполнением
- Формат ответа должен соответствовать спецификации MCP

#### 1.3 Проблемы сетевой интеграции: Docker, WSL и проброс портов (5 минут)

**Проблема:** n8n работает в Docker контейнере, а 1С база опубликована в WSL. Как обеспечить связь между ними?

**Решение через проброс портов:**

1. **Настройка на хосте Windows:**
   - Использование скрипта `setup-wsl-port-forward.ps1`
   - Проброс порта из WSL (например, 18080) на хост (localhost:18080)
   - Настройка маршрутизации через `add-wsl-route.ps1`

2. **Настройка в docker-compose.yml:**
   ```yaml
   n8n:
     extra_hosts:
       - "host.docker.internal:host-gateway"
       - "BXZ-NOTE.WSL:172.17.238.77"
   ```

3. **Использование wsl-proxy контейнера:**
   ```yaml
   wsl-proxy:
     image: alpine/socat:latest
     command: TCP-LISTEN:8080,fork,reuseaddr TCP-CONNECT:host.docker.internal:18080
     ports:
       - "8080:8080"
   ```

**Важно:**
- Для production использовать более безопасные методы
- Рассмотреть использование VPN или защищенных туннелей
- Ограничить доступ по IP адресам

### Блок 2: LangGraph - фреймворк для сложных агентов (35 минут)

#### 2.1 Что такое LangGraph и зачем он нужен (10 минут)

**LangGraph** — это низкоуровневый фреймворк для построения долгоживущих, stateful агентов с графовой архитектурой.

**Ключевые отличия от простых агентов:**

1. **Графовая архитектура**
   - Агент представляется как граф состояний
   - Узлы графа — это функции (nodes)
   - Ребра графа — это переходы между состояниями (edges)
   - Позволяет создавать сложные логические потоки

2. **Stateful агенты**
   - Агент сохраняет состояние между вызовами
   - Может возобновить работу после сбоя
   - Поддерживает долгоживущие сессии

3. **Durable execution**
   - Агент может работать часами или днями
   - Автоматическое сохранение состояния (checkpointing)
   - Возобновление с места остановки при сбое

**Когда использовать LangGraph:**
- Сложные многошаговые задачи
- Агенты, требующие сохранения состояния
- Долгоживущие процессы
- Когда нужен контроль над потоком выполнения

**Когда НЕ использовать:**
- Простые одношаговые задачи
- Статичные workflow без адаптации
- Когда достаточно n8n или простых агентов

---

#### 2.2 Архитектура LangGraph (10 минут)

**Основные концепции:**

1. **State (Состояние)**
   - Хранит всю информацию о текущем состоянии агента
   - Может включать: сообщения, результаты вызовов, промежуточные данные
   - Обновляется на каждом шаге

2. **Nodes (Узлы)**
   - Функции, которые выполняют действия
   - Могут вызывать LLM, инструменты, обрабатывать данные
   - Принимают state, возвращают обновленный state

3. **Edges (Ребра)**
   - Определяют переходы между узлами
   - Могут быть условными (conditional edges)
   - Определяют следующий шаг на основе состояния

4. **Checkpoints (Точки сохранения)**
   - Сохраняют состояние агента
   - Позволяют возобновить выполнение
   - Могут храниться в БД или памяти

**Пример простого графа:**

```
[START] 
    ↓
[Получить запрос пользователя]
    ↓
[Анализ запроса] → (нужны данные из 1С?)
    ↓ да                    ↓ нет
[Вызов MCP tool]      [Генерация ответа]
    ↓
[Обработка результата]
    ↓
[Формирование ответа]
    ↓
[END]
```

**Код примера:**

```python
from langgraph.graph import StateGraph, END
from typing import TypedDict, Annotated
import operator

# Определение состояния
class AgentState(TypedDict):
    messages: Annotated[list, operator.add]
    user_query: str
    mcp_result: dict | None

# Создание графа
graph = StateGraph(AgentState)

# Добавление узлов
graph.add_node("analyze_query", analyze_query_node)
graph.add_node("call_mcp", call_mcp_node)
graph.add_node("generate_response", generate_response_node)

# Добавление переходов
graph.set_entry_point("analyze_query")
graph.add_conditional_edges(
    "analyze_query",
    should_call_mcp,  # Функция, определяющая следующий шаг
    {
        "yes": "call_mcp",
        "no": "generate_response"
    }
)
graph.add_edge("call_mcp", "generate_response")
graph.add_edge("generate_response", END)

# Компиляция графа
app = graph.compile()
```

#### 2.3 Ключевые возможности LangGraph (10 минут)

**1. Durable Execution (Долговечное выполнение)**

Агент может работать долго и переживать сбои:

```python
from langgraph.checkpoint.memory import MemorySaver

# Создание хранилища для checkpoint
memory = MemorySaver()

# Компиляция с checkpoint
app = graph.compile(checkpointer=memory)

# Запуск с сохранением состояния
config = {"configurable": {"thread_id": "user-123"}}
result = app.invoke(
    {"messages": [{"role": "user", "content": "Какие продажи за январь?"}]},
    config=config
)

# При сбое можно возобновить:
# app.invoke(..., config=config)  # Продолжит с места остановки
```

**2. Human-in-the-loop (Вмешательство человека)**

Агент может запрашивать подтверждение:

```python
from langgraph.prebuilt import interrupt_before

# Добавление точки прерывания
graph.add_node("approve_action", approve_action_node)
graph = interrupt_before(graph, "approve_action")

# При выполнении агент остановится и запросит подтверждение
```

**3. Comprehensive Memory (Комплексная память)**

- **Краткосрочная память:** в контексте LLM (последние N сообщений)
- **Долгосрочная память:** в векторной БД или графе знаний
- **Рабочая память:** в состоянии агента (state)

**4. Debugging with LangSmith**

- Визуализация выполнения графа
- Трассировка состояния на каждом шаге
- Метрики производительности
- Отладка ошибок

**5. Production-ready Deployment**

- Масштабирование через LangSmith Deployment
- Мониторинг и логирование
- Управление версиями агентов
- A/B тестирование

#### 2.4 LangGraph vs простые агенты (5 минут)

**Сравнение подходов:**

| Аспект | Простые агенты (n8n) | LangGraph |
|--------|----------------------|-----------|
| Сложность настройки | Низкая | Средняя-высокая |
| Визуальное программирование | Да | Нет (код) |
| Stateful агенты | Ограниченно | Полная поддержка |
| Durable execution | Нет | Да |
| Human-in-the-loop | Ограниченно | Полная поддержка |
| Масштабирование | Среднее | Высокое |
| Отладка | Базовая | Продвинутая (LangSmith) |
| Подходит для | Простые задачи | Сложные долгоживущие процессы |

**Когда использовать каждый подход:**

**Простые агенты (n8n):**
- Быстрое прототипирование
- Простые автоматизации
- Визуальное программирование важно
- Не нужна долгоживущая сессия

**LangGraph:**
- Сложные многошаговые задачи
- Долгоживущие процессы (часы/дни)
- Критически важна надежность
- Нужен контроль над потоком выполнения
- Требуется отладка и мониторинг

### Блок 3: Интеграция LangGraph с MCP сервером 1С (25 минут)

#### 3.1 Создание агента для извлечения данных из 1С (15 минут)

**Задача:** Создать LangGraph агента, который:
- Получает запросы пользователя на естественном языке
- Определяет, какие данные нужны из 1С
- Вызывает соответствующий MCP tool
- Обрабатывает результат
- Формирует понятный ответ

**Архитектура агента:**

```python
from langgraph.graph import StateGraph, END
from langgraph.prebuilt import ToolNode
from langchain_openai import ChatOpenAI
from langchain_core.messages import HumanMessage, AIMessage
import httpx

# Определение состояния
class OneCAgentState(TypedDict):
    messages: Annotated[list, operator.add]
    user_query: str
    mcp_tools: list
    mcp_result: dict | None
    final_answer: str | None

# Инициализация LLM
llm = ChatOpenAI(
    model="gpt-4o-mini",
    base_url="http://host.docker.internal:1234/v1",
    api_key="lm-studio"
)

# Узел анализа запроса
def analyze_query_node(state: OneCAgentState):
    """Анализирует запрос и определяет нужный MCP tool"""
    
    query = state["user_query"]
    
    # Определяем, какой tool нужен
    if "продажи" in query.lower() or "выручка" in query.lower():
        tool_name = "get_sales"
    elif "метаданные" in query.lower() or "структура" in query.lower():
        tool_name = "get_metadata"
    else:
        tool_name = None
    
    return {
        "mcp_tools": [tool_name] if tool_name else [],
        "messages": [AIMessage(content=f"Определен tool: {tool_name}")]
    }

# Узел вызова MCP
def call_mcp_node(state: OneCAgentState):
    """Вызывает MCP tool из 1С"""
    
    tool_name = state["mcp_tools"][0]
    
    # Извлекаем параметры из запроса пользователя
    # (упрощенный пример, в реальности нужен более умный парсинг)
    params = extract_params_from_query(state["user_query"])
    
    # Вызов MCP сервера
    response = httpx.post(
        "http://host.docker.internal:18080/база/hs/mcp/tools/call",
        json={
            "name": tool_name,
            "arguments": params
        },
        timeout=30.0
    )
    
    result = response.json()
    
    return {
        "mcp_result": result,
        "messages": [AIMessage(content=f"Получен результат от 1С: {result}")]
    }

# Узел генерации ответа
def generate_response_node(state: OneCAgentState):
    """Генерирует финальный ответ пользователю"""
    
    query = state["user_query"]
    mcp_result = state["mcp_result"]
    
    # Формируем промпт для LLM
    prompt = f"""
    Пользователь спросил: {query}
    
    Данные из 1С:
    {mcp_result}
    
    Сформируй понятный ответ на русском языке на основе полученных данных.
    """
    
    response = llm.invoke([HumanMessage(content=prompt)])
    
    return {
        "final_answer": response.content,
        "messages": [AIMessage(content=response.content)]
    }

# Создание графа
graph = StateGraph(OneCAgentState)

graph.add_node("analyze_query", analyze_query_node)
graph.add_node("call_mcp", call_mcp_node)
graph.add_node("generate_response", generate_response_node)

graph.set_entry_point("analyze_query")

graph.add_conditional_edges(
    "analyze_query",
    lambda state: "call_mcp" if state["mcp_tools"] else "generate_response",
    {
        "call_mcp": "call_mcp",
        "generate_response": "generate_response"
    }
)

graph.add_edge("call_mcp", "generate_response")
graph.add_edge("generate_response", END)

# Компиляция
app = graph.compile()
```

**Использование:**

```python
result = app.invoke({
    "user_query": "Какие продажи были в январе 2024?",
    "messages": []
})

print(result["final_answer"])
```

#### 3.2 Обработка ошибок и retry логика (5 минут)

**Важно:** Агенты должны быть устойчивы к ошибкам.

**Обработка ошибок в узлах:**

```python
def call_mcp_node(state: OneCAgentState):
    """Вызывает MCP tool с обработкой ошибок"""
    
    try:
        tool_name = state["mcp_tools"][0]
        params = extract_params_from_query(state["user_query"])
        
        response = httpx.post(
            "http://host.docker.internal:18080/база/hs/mcp/tools/call",
            json={"name": tool_name, "arguments": params},
            timeout=30.0
        )
        
        if response.status_code != 200:
            raise Exception(f"MCP сервер вернул ошибку: {response.status_code}")
        
        result = response.json()
        return {"mcp_result": result}
        
    except httpx.TimeoutException:
        return {
            "mcp_result": None,
            "messages": [AIMessage(content="Таймаут при обращении к 1С. Попробуйте позже.")]
        }
    except Exception as e:
        return {
            "mcp_result": None,
            "messages": [AIMessage(content=f"Ошибка при обращении к 1С: {str(e)}")]
        }
```

**Особенности:**

- Может искать объекты по описанию
- Показывает связи между объектами
- Помогает в разработке

**Пример использования:**

```
Пользователь: "Найди все документы, связанные со справочником Контрагенты"
Агент:
1. Получает метаданные через MCP
2. Анализирует связи
3. Возвращает список документов
```

#### 4.3 Мультиагентная система для 1С (6 минут)

**Концепция:** Несколько специализированных агентов работают вместе.

**Архитектура:**

```
[Роутер-агент]
    ├──→ [Агент продаж] → MCP: get_sales
    ├──→ [Агент метаданных] → MCP: get_metadata
    ├──→ [Агент документов] → MCP: search_documents
    └──→ [Агент аналитики] → MCP: get_reports
```

**Реализация в LangGraph:**

```python
# Роутер определяет, какой агент нужен
def router_node(state):
    query = state["user_query"]
    
    if "продажи" in query:
        return {"target_agent": "sales_agent"}
    elif "метаданные" in query:
        return {"target_agent": "metadata_agent"}
    # ...

# Каждый агент - отдельный подграф
sales_graph = create_sales_agent_graph()
metadata_graph = create_metadata_agent_graph()

# Главный граф с роутингом
main_graph = StateGraph(MainState)
main_graph.add_node("router", router_node)
main_graph.add_node("sales_agent", sales_graph)
main_graph.add_node("metadata_agent", metadata_graph)
```

### Блок 5: Best practices и рекомендации (15 минут)

#### 5.1 Разработка MCP инструментов (5 минут)

**Рекомендации:**

1. **Идемпотентность**
   - Повторный вызов с теми же параметрами дает тот же результат
   - Не должно быть побочных эффектов

2. **Валидация параметров**
   - Проверяйте все параметры перед выполнением
   - Возвращайте понятные ошибки

3. **Обработка ошибок**
   - Всегда обрабатывайте исключения
   - Возвращайте структурированные ошибки

4. **Производительность**
   - Ограничивайте количество возвращаемых записей
   - Используйте пагинацию для больших выборок
   - Оптимизируйте запросы к БД

5. **Документация**
   - Подробно описывайте параметры
   - Указывайте формат данных
   - Приводите примеры использования

---

#### 5.2 Разработка LangGraph агентов (5 минут)

**Рекомендации:**

1. **Проектирование графа**
   - Начинайте с простого графа
   - Постепенно усложняйте
   - Используйте условные переходы для гибкости

2. **Управление состоянием**
   - Храните только необходимые данные
   - Используйте типизацию (TypedDict)
   - Очищайте состояние при необходимости

3. **Обработка ошибок**
   - Добавляйте узлы для обработки ошибок
   - Используйте retry логику
   - Логируйте ошибки для отладки

4. **Тестирование**
   - Тестируйте каждый узел отдельно
   - Тестируйте весь граф
   - Используйте mock данные для MCP

5. **Мониторинг**
   - Используйте LangSmith для отладки
   - Логируйте важные события
   - Отслеживайте производительность

---

#### 5.3 Безопасность и производительность (5 минут)

**Безопасность:**

1. **Аутентификация**
   - Используйте токены для MCP серверов
   - Ограничивайте доступ по IP
   - Не используйте безпарольный вход в production

2. **Валидация данных**
   - Проверяйте все входные данные
   - Защищайтесь от SQL injection
   - Ограничивайте размер запросов

3. **Логирование**
   - Логируйте все обращения к MCP
   - Не логируйте чувствительные данные
   - Используйте структурированное логирование

**Производительность:**

1. **Кеширование**
   - Кешируйте результаты MCP вызовов
   - Используйте TTL для кеша
   - Инвалидируйте кеш при изменениях

2. **Асинхронность**
   - Используйте async вызовы где возможно
   - Параллелизуйте независимые операции
   - Используйте пулы соединений

3. **Оптимизация LLM**
   - Ограничивайте размер контекста
   - Используйте streaming для длинных ответов
   - Кешируйте промпты

## Резюме лекции (5 минут)

### Ключевые моменты:

1. **MCP (Model Context Protocol):**
   - Стандартизированный протокол для инструментов
   - Состоит из tools, resources и prompts
   - Упрощает интеграцию агентов с внешними системами
   - MCP сервер для 1С позволяет создавать инструменты внутри системы

2. **LangGraph:**
   - Фреймворк для создания сложных stateful агентов
   - Графовая архитектура с узлами и переходами
   - Поддержка durable execution и checkpointing
   - Human-in-the-loop и комплексная память
   - Интеграция с LangSmith для отладки

3. **Интеграция с 1С:**
   - MCP серверы позволяют агентам работать с данными 1С
   - LangGraph агенты могут вызывать MCP tools
   - Обработка ошибок и retry логика критически важны
   - Память агента позволяет сохранять контекст между запросами

4. **Практические применения:**
   - Агенты для анализа данных из 1С
   - Агенты для работы с метаданными
   - Мультиагентные системы для сложных задач

5. **Best practices:**
   - Идемпотентность и валидация для MCP tools
   - Постепенное усложнение графов в LangGraph
   - Безопасность и производительность в production
