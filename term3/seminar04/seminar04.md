# Семинар 4: Отложенные обработчики обновления ИБ (БСП)

**Продолжительность:** ~90 минут  
**Целевая аудитория:** Студенты 3 курса, прошедшие курс разработки на платформе 1С:Предприятие (1–2 семестры)  
**Курс:** Библиотека стандартных подсистем  
**Источник занятия:** `БСП_06_Отложенные_операции_2026-03-20.txt` (демо: документ **«Реализация товаров и услуг»**, переход от **монопольного** обработчика к **отложенному многопоточному**)

## Цель семинара

Закрепить **рекомендации** по обработчикам обновления ИБ: серверный код, **идемпотентность** (в т.ч. при сбоях фоновых заданий и повторном запуске), отсутствие скрытых зависимостей между обработчиками (**порядок вызова не гарантирован**). Разобрать задачи фонового этапа после запуска пользователей: **блокировка** ещё не мигрированных объектов, **многопоточность** с ограничением по **числу ядер**, **блокировка данных** на время обработки объекта в транзакции. Практически оформить **отложенный** обработчик через обработку **«Описание обработчиков обновления»** (шаблоны в поставке БСП): методы **заполнение данных**, **обработка данных**, **проверка**; **читаемые** и **изменяемые** объекты; регистрация на плане обмена **«Обновление информационной базы»** с запретом **авторегистрации**; **`ОтметитьКОбработке`** (порядок **основных параметров** и данных), **`ДанныеДляОбновленияИнформационнойБазыВМногопоточномОбработчике`**, транзакция **`Попытка`/`Исключение`**, **`БлокировкаДанных`**, **`ОтметитьВыполнениеОбработки`**, **`ОбновлениеИнформационнойБазы.ЗаписатьОбъект`**, **`ОбработкаДанныхЗавершена`**; при ошибке в многопоточном режиме — **журнал регистрации** без «роняния» всего конвейера. Связать с **режимом выполнения отложенных обработчиков** подсистемы (**параллельно** vs последовательно), отладкой (фактически **однопоточно** при отладке из конфигуратора), мониторингом **Администрирование → Обслуживание → Обновление приложения → Результаты**. По документации БСП — **блокировка необработанных данных** в форме и **`ПередЗаписью`**, контроль **зацикливания**.

## Необходимые предзнания

- Лекция по **обновлению ИБ** ([lection04.md](../lection04/lection04.md)): виды обработчиков, модуль **«Обновление информационной базы»** библиотеки, версии подсистемы
- Монопольный обработчик заполнения складов в ТЧ **«Реализация товаров и услуг»** (курс): отбор «ещё не обработано», пустой склад в ТЧ при заполненной «старой» логике
- Запросы, **планы обмена**, **управляемые блокировки**, транзакции **Попытка / Исключение** по стандартам разработки

## План семинара (~90 минут)

> **Примечание:** Время зависит от объёма данных в учебной базе, режима работы (файл / сервер) и повторения шагов за преподавателем.

### Блок 1: Правила обработчиков и риски «длинного» обновления (12 минут)

#### 1.1 Общие рекомендации

- Код обновления выполняется на **сервере**: **нет** интерактива с пользователем.
- При **критической** ошибке — **проброс исключения**; старт системы может быть заблокирован до исправления.
- Обработчик должен быть **рассчитан на повторное выполнение** без **дублирования** эффекта в данных (**идемпотентность**).

#### 1.2 Учёт уже изменённых пользователем данных

- В выборку попадают только объекты, которым **нужна** миграция (пример: старые документы до перехода на новую модель складов в ТЧ).
- Не перетирать доработки пользователя после обновления: в запросе отсекать документы, где миграция уже не нужна (например, склад в ТЧ заполнен — порция обработана или пользователь внёс данные осознанно).

#### 1.3 Независимость обработчиков

- Обработчиков может быть **много**; **порядок вызова не детерминирован** — нельзя строить логику «сначала А, потом Б». При зависимости — **объединить** в один обработчик.

#### 1.4 Зачем отложенная обработка и порции

- После запуска пользователей нужно **не дать редактировать** документы, которые ещё не прошли миграцию (см. блокировка в интерфейсе и план обмена — блок 6).
- Обработку дробят на **фоновые задания** и **потоки**. По записи занятия: число потоков выполнения **не должно превышать** число **ядер процессоров** сервера 1С:Предприятия (иначе риск перегруза без выигрыша).
- Взяв объект в обработку в многопотоке, нужно **заблокировать** его на время работы (логика может быть **долгой**) и после успеха **явно** отметить результат — для **идемпотентности** при падении фоновых заданий или перезапуске.
- Крупный «архив» не обязан обрабатываться в монополии: его выносят в фон; при этом важны **порции** и рекомендации из **документации БСП** (на занятии: ориентир **~1000** объектов в порции; при **слишком больших** порциях — порядка **10 000** — фоновое задание может «захлебнуться» из‑за длительной обработки).
- Два уровня **приоритета** (см. параметры выборки): что важнее — **работа пользователей** или **скорость обработки данных**; плюс отдельно задаётся упорядочивание (например, **дата по убыванию** — сначала «свежие» документы).

### Блок 2: Настройка многопоточного обновления БСП (8 минут)

- В **переопределяемом** модуле настроек обновления ИБ (в демо — контекст **«демонстрационное обновление ИБ БСП»** / аналог в вашей поставке) задаются признак **многопоточности** и **количество потоков** (на курсе встречался пример **4**; сверять с **ИТС** и своей конфигурацией).
- Фактическое число потоков может переопределяться **константой** вида **«Количество потоков обновления информационной базы»** (имя — по **вашей** редакции БСП).
- **Критично для многопоточного отложенного обработчика:** у **подсистемы** (в демо — «Курс разработки») свойство **режим выполнения отложенных обработчиков** должно быть **параллельно**. Если подсистема в режиме **последовательного** выполнения обработчиков, платформа выдаёт ошибку о **недопустимом** свойстве «многопоточный» у отложенного обработчика.

### Блок 3: Описание отложенного обработчика в конфигураторе (15 минут)

#### 3.1 Где задаётся таблица обработчиков

- В модуле библиотеки с **`ПриДобавленииОбработчиковОбновления`** формируется таблица через **`ОбновлениеИнформационнойБазы.НоваяТаблицаОбработчиковОбновления`** (программный интерфейс БСП — по справке редакции).

#### 3.2 Обработка «Описание обработчиков обновления»

- Шаблон обработки поставляется с БСП (на занятии путь к типовой установке: каталог **templates**, **1С**, **Библиотека стандартных подсистем** — ориентир для поиска материалов).
- В режиме **1С:Предприятие** использовать обработку **«Описание обработчиков обновления»**: список уже описанных обработчиков; добавление строки с **отложенным** режимом выполнения, **идентификатором**, **объектом метаданных** (документ **«Реализация товаров и услуг»**). Платформа генерирует заготовки **экспортных** процедур — их реализуют в **модуле менеджера** документа.
- В документации БСП поясняется, почему нельзя «просто получить объект и записать»: рекомендуется следовать **API подсистемы обновления ИБ** (несколько вызовов вместо «голой» записи).

#### 3.3 Ключевые поля мастера

- **Объект метаданных** — обрабатываемый тип; если тип не виден в списке, проверить **справочник идентификаторов объектов метаданных** (демо) и заполнение обязательных полей.
- **Идентификатор** — для сопоставления при переименованиях/правках процедуры.
- **Процедуры**: **заполнение данных** (регистрация к обработке), **обработка данных** (порция в потоке), **проверка** — для отложенного обработчика процедура **проверки** должна быть **описана** (хотя бы минимально), иначе обработчик не запишется.
- **Заполнение данных** — формирование и регистрация ссылок через **`ОтметитьКОбработке`** (см. блок 4).
- **Версия** при переходе на которую срабатывает обработчик (пример с занятия: **1.0.3.1**); **начальное заполнение** в учебном сценарии отключали (**не нужно**).

#### 3.4 Режимы и приоритеты

- Режим выполнения обработчика: **отложенный**; флаг **многопоточный** (если позволяет режим **подсистемы** — см. блок 2).
- **Порядок** обработки (в демо — **обычный**): в сочетании с параметрами выборки задаёт приоритет очередей (**актуальные** данные vs остальное — см. **параметры выборки актуальных данных**).

#### 3.5 Читаемые и изменяемые объекты

- **Изменяемый объект** — документ, по которому идёт миграция (нужна **блокировка на запись**).
- **Читаемый объект** (если указывается отдельно) — по смыслу **блокировка чтения**; для учебного сценария акцент на **изменяемом** объекте.
- Обязательно заполнить **текстовое описание** назначения обработчика (например, перенос склада из шапки в табличную часть) — иначе отложенный обработчик **не сохраняется** (проверка в конфигураторе).
- **«Показать код обработчиков»** — перенести фрагмент в **общий модуль обновления вашей библиотеки** (`ПриДобавленииОбработчиковОбновления`), а не в модуль менеджера документа; в модуле менеджера — только **реализация** процедур, на которые ссылается описание.
- Чтобы повторить сценарий после монопольного прогона, **очистить** тестовые данные / откатить результат монопольного переноса там, где это нужно для демонстрации отложенного обработчика.

### Блок 4: «Заполнение данных» — регистрация на плане обмена (18 минут)

#### 4.1 Сигнатура

- Экспортная процедура с единственным параметром **`Параметры`** (структура/контекст БСП для обновления).

#### 4.2 Вызов `ОтметитьКОбработке`

- **`ОбновлениеИнформационнойБазы.ОтметитьКОбработке`**: соблюдать **порядок и состав** параметров по **комментариям к методу** и **ИТС**. На занятии была ошибка **«недостаточно фактических параметров»**: перепутали блоки — сначала идут **основные параметры** (контекст обновления), **затем** данные для регистрации (массив ссылок и т.д. — см. синтакс-помощник).
- Для регистров с отбором по регистраторам — **`ОтметитьРегистраторыКОбработке`** и др.; в учебном примере — **ссылки на документы**.

#### 4.3 Данные к регистрации

- Запрос — как в **монопольном** обработчике; для передачи в **`ОтметитьКОбработке`** удобно **`Выгрузить()`** + **`ВыгрузитьКолонку("Ссылка")`**.
- **Из речи:** ориентир размера порции **~1000** объектов; при слишком больших порциях (упоминался порядок **10 000**) фоновое задание может выполняться неприемлемо долго.

#### 4.4 Параметры выборки (`ПараметрыВыборки`)

- Ориентиры из справки: **основные параметры**, **отметки**, **параметры выборки**, **дополнительные параметры выборки** (в т.ч. для **многопоточной** обработки).
- **Полное имя объекта** — **`ПустаяСсылка().Метаданные().ПолноеИмя()`** в **модуле менеджера** документа.
- **Упорядочение при работе пользователей** — массив (в демо: **дата убыв.**): сначала **свежие** документы.
- **Поля упорядочения при обработке данных** — отдельный массив для второго уровня приоритета (в справке различают сценарии «важнее не мешать пользователям» и «важнее догнать обработку»).
- **Способ выборки** — строка; для документов в демо — **по ссылкам**; для регистров — иные варианты (**регистраторы** и т.д.).

#### 4.5 Актуальные данные

- Использовать **`ПараметрыВыборкиАктуальныхДанных`** (имя уточнить в **вашей** редакции БСП): **поле отбора**, **вид сравнения**, **значение**. Пример с занятия: **Дата**, **Больше**, значение — **начало месяца** по **дате сеанса** — в приоритет попадают документы **новее** этой границы.

#### 4.6 План обмена

- План **«Обновление информационной базы»**: при необходимости **снять с поддержки**, в **состав** включить документ **«Реализация товаров и услуг»**.
- **Авторегистрацию отключить** — регистрация только из **кода** миграции.

### Блок 5: «Обработать данные…» — порция, транзакция, блокировка, запись (22 минуты)

#### 5.1 Получение порции

- **`ОбновляемыеДанные = ОбновлениеИнформационнойБазы.ДанныеДляОбновленияИнформационнойБазыВМногопоточномОбработчике(Параметры)`**  
  Возвращается **таблица значений**; для документа — колонки в том числе **Ссылка**, **Дата** (для регистров — измерения).

#### 5.2 Цикл по строкам

- Итерировать **строки таблицы**, не «сами ссылки» вне контекста строки.

#### 5.3 Транзакция и блокировка

- **`НачатьТранзакцию`**, тело — в **`Попытка` / `Исключение` / `КонецПопытки`** по **стандартам разработки**: в **`Исключение`** — **`ОтменитьТранзакцию`**, далее по методичке — проброс или обработка ошибки.
- **`БлокировкаДанных`**: режим **исключительный**, полное имя таблицы + **значение = ссылка** на документ. Пример кода — из **синтакс-помощника**. Блокировка держится **до фиксации или отмены транзакции**; «снять блокировку отдельно» нельзя.
- В **`Параметры`** при необходимости можно добавлять **свои поля** для обмена данными между потоками (на занятии не углублялись).

#### 5.4 Удалённый или недоступный объект

- Если **`ПолучитьОбъект()`** вернул **`Неопределено`** (битая ссылка, объект удалён) — **`ОтметитьВыполнениеОбработки`** по ссылке, **зафиксировать** транзакцию, **перейти** к следующей строке — чтобы обработчик **не зацикливался** на «мёртвых» ссылках.

#### 5.5 Прикладная логика и запись

- Логика миграции как в монопольном примере (перенос склада в ТЧ); флаг **`ОбъектИзменен`** — если хотя бы одна строка изменена.
- Если объект **изменён** — **`ОбновлениеИнформационнойБазы.ЗаписатьОбъект`** (в речи — «записать объект» API обновления ИБ: **`ОбменДанными.Загрузка`**, без автoregистрации на обмене — см. лекцию по обновлению ИБ).
- Если **не изменён** — всё равно **`ОтметитьВыполнениеОбработки`**, чтобы снять с очереди.

#### 5.6 Завершение порции

- В конце — **`ОбновлениеИнформационнойБазы.ОбработкаДанныхЗавершена`**: передать **очередь**, **полное имя метаданных** и актуальные **`Параметры`** по шаблону БСП.

#### 5.7 Ошибки, зацикливание, журнал

- В **многопоточном** обработчике **нельзя** бездумно **`ВызватьИсключение`** в блоке обработки одного объекта — можно «уронить» весь режим; ошибку нужно **зарегистрировать** (**`ЗаписьЖурналаРегистрации`**: событие связано с **обновлением ИБ**, уровень **предупреждение**, в данные — метаданные/ссылка, текст — **`ОбработкаОшибок.ПодробноеПредставлениеОшибки`**).
- По документации БСП (**контроль зацикливания**): большое число итераций и **пустые** прогоны подряд — сигнал проблемы в логике отбора/отметки.

### Блок 6: Блокировка необработанных данных в интерфейсе (8 минут)

- Раздел документации БСП: **«Блокировка необработанных данных в пользовательском интерфейсе»**.
- Две вставки кода:
  1. В форме документа — обработчик **`ПриСозданииНаСервере`**.
  2. В **модуле объекта** — **`ПередЗаписью`**.
- Цель: пользователь не может **редактировать** документы, пока они не сняты с плана обмена обновления ИБ.

### Блок 7: Проверка внедрения, данные для повторного прогона, отладка (12 минут)

#### 7.1 Версия и «Проверка внедрения БСП»

- Поднять **версию** подсистемы в модуле обновления так, чтобы сработала нужная ветка обработчиков (на занятии отдельно следили, чтобы не «перепрыгнуть» на старый монопольный сценарий: например, целевая **3.1** для отложенного при **2.1** на промежуточном шаге — ориентир из речи, сверять с вашей нумерацией).
- Отчёт **«Проверка внедрения БСП»** и контроль состава плана обмена **«Обновление информационной базы»** — регулярная практика.

#### 7.2 Повтор демонстрации на учебных документах

- Если данные уже обработаны монопольным обработчиком — подготовить тестовые документы заново (**очистить** ТЧ / восстановить сценарий «шапка заполнена — строки пустые»), чтобы отложенный обработчик было что мигрировать.

#### 7.3 Отладка

- Запуск с **обновлением ИБ** и **режимом отладки** (**сервис → параметры**); точка останова в процедуре **обработки данных** модуля менеджера.
- **Важно:** при включённой **отладке из конфигуратора** фактически выполняется **однопоточный** режим — даже если в обработчике указано **многопоточное** выполнение (как в реплике с занятия).
- Ошибки в форме **дополнительной обработки данных** (например, поле **обновляемые данные** не найдено) часто связаны с тем, что код выполняется **не в многопоточной** ветке — проверить настройки обработчика и подсистемы.

#### 7.4 Клиент-сервер и мониторинг

- **Администрирование → Обслуживание → Обновление приложения → Результаты** (формулировки меню — по вашей версии интерфейса): видно выполнение **дополнительной обработки данных**, перенос склада и т.д.; можно переключить **приоритет** (в речи — «переключить на обработку данных» vs приоритет работы пользователя).
- На время отладки полезно учитывать **регламентные задания** и фон, чтобы не смешивать сценарии.

#### 7.5 Процедура «Проверка»

- Если платформа сообщает, что **не заполнена процедура проверки** — добавить процедуру с корректной **сигнатурой** (параметры проверки по справке); пустая реализация допустима для прохождения проверки конфигурации, но сценарий нужно согласовать с **ИТС**.

## Резюме семинара (5 минут)

### Ключевые моменты

1. Обработчик обновления — **серверный**, **идемпотентный**; **нельзя** опираться на порядок вызова разных обработчиков.
2. Фон после запуска пользователей: **блокировки**, **потоки ≤ ядер**, **блокировка данных** на объект в транзакции; крупные объёмы — **порции** (~**1000**, осторожно с **10 000**).
3. Подсистема: **режим отложенных обработчиков — параллельно**; иначе **многопоточный** флаг у отложенного обработчика **недопустим**.
4. Три метода: **заполнение данных** (**`ОтметитьКОбработке`** — не перепутать порядок аргументов); **обработка данных** — **`ДанныеДляОбновленияИнформационнойБазыВМногопоточномОбработчике`**; **проверка** — должна быть описана.
5. План **«Обновление ИБ»**: документ в составе, **авторегистрация выключена**; **читаемый** vs **изменяемый** объект; описание обработчика в конфигураторе **обязательно**.
6. Порция: **транзакция**, **`БлокировкаДанных`**, **`ОтметитьВыполнениеОбработки`**, **`ЗаписатьОбъект`**, **`ОбработкаДанныхЗавершена`**; при ошибке по объекту — **журнал**, не «ронять» весь многопоток.
7. Отладка из конфигуратора — фактически **однопоток**; мониторинг — **Обновление приложения → Результаты**; блокировка в **форме** и **`ПередЗаписью`** по ИТС.

---

### Практические рекомендации

1. Имена методов и точные сигнатуры сверять с **синтакс-помощником** и **ИТС** для **вашей** редакции БСП; при работе с БСП **не стесняться документации** — в т.ч. большие разделы про обновление ИБ и многопоточность.
2. После правок плана обмена и обработчиков — **«Проверка внедрения БСП»** и контрольная **выгрузка** учебной базы.
3. На сервере при миграции — окно обслуживания, **РЗ**, мониторинг **журнала регистрации** и **Администрирование → Обслуживание → Обновление приложения → Результаты**.
