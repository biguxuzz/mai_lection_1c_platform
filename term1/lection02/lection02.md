## План лекции (110 минут)

### Блок 1: Константы (5 минут)

#### 1.1 Назначение констант (3 минуты)

**Константы** в платформе 1С — это реквизиты самой конфигурации, предназначенные для хранения условно постоянных величин.

**Отличия от программирования:**
- В обычном программировании константа — это постоянное значение, обозначенное в коде
- В 1С константа — это объект метаданных, значение которого хранится в базе данных
- Значения констант можно изменять в процессе эксплуатации системы

**Примеры использования:**
- Основной вид номенклатуры для работы
- Адрес внешнего сервиса для интеграции
- Основная организация
- Настройки ведения учета

**Важно:** Изменение значений констант может привести к изменению логики бизнес-решения.

---

#### 1.2 Типы данных констант (2 минуты)

В константах можно хранить:
- Практически любой ссылочный тип
- Хранилище значений (включая бинарные файлы)
- Ссылки на значения, которые затем используются в коде

**Пример:**
```1c
// Получение значения константы
ОсновнаяОрганизация = Константы.ОсновнаяОрганизация.Получить();
```

---

#### 1.3 Примеры с лекции (настраиваемое поведение без релиза)

- **Интеграция:** URL/endpoint внешнего сервиса в константе — при смене адреса меняется фактическая «точка» обращения системы без пересборки конфигурации.
- **Одна vs несколько организаций:** булева константа вида «ведём учёт по нескольким организациям» — при `Ложь` справочник организаций может использоваться в данных, но **лишние поля/шаги** в формах не показываются пользователю.
- **Ордерная схема на складе:** флаг включения цепочки «заказ → резерв → расходный ордер» vs «упрощённый ларёк», где достаточно одной реализации — константа отключает ненужные документы и команды.
- **УСН и НДС:** организация на упрощёнке — константа «учитывать НДС» в `Ложь`; при выходе на обычный режим — переключение флага и появление НДС в документах **без** смены релиза как такового (настройка жизненного цикла ИБ).
- **Ограничение гибкости:** для большого числа **настроечных** переключателей удобнее **план видов характеристик** и пользовательские значения, чем размножение констант (см. блок про ПВХ и финальные тезисы лекции про доработки).

### Блок 2: Справочники (30 минут)

#### 2.1 Назначение и основные свойства (10 минут)

**Справочники** нужны для хранения каталожных (перечислимых) данных.

**Примеры справочников:**
- Контрагенты
- Договоры
- Сотрудники
- Физические лица
- Валюты
- Номенклатура

**Основные реквизиты справочника:**

1. **Код** — короткая идентификация элемента
   - Обычно присваивается автоматически
   - Можно влиять на формирование кода через события
   - В распределенных базах помогает определить узел создания элемента (префикс)

2. **Наименование** — человекочитаемое представление
   - Для контрагентов: может быть короткое наименование ("Behring Pro")
   - Полное наименование хранится в отдельном реквизите ("Общество с ограниченной ответственностью...")

**Важно:** Можно отказаться от кода и наименования, справочник будет идентифицироваться по другим реквизитам.

---

#### 2.2 Табличные части справочников (5 минут)

**Табличные части** используются для хранения множественных записей одного типа.

**Примеры использования:**
- Контактная информация контрагента:
  - Юридический адрес
  - Физический адрес
  - Почтовый адрес
- Спецификация номенклатуры (элементы, из которых состоит товар)

**Особенности:**
- Одинаковый тип данных в каждой строке
- Количество записей может варьироваться
- Табличные части также используются в документах

---

#### 2.3 Предопределенные элементы (5 минут)

**Предопределенные элементы** — это элементы справочника, созданные на уровне конфигурации и доступные в коде.

**Проблема без предопределенных элементов:**
- Обращение к объектам по имени из кода — плохая практика
- Имя может измениться
- Пользователь может удалить элемент
- Возникают проблемы при эксплуатации

**Решение:**
Создать предопределенный элемент на уровне справочника (например, "Розничный покупатель") и обращаться к нему в коде. Пользователь может **переименовать** элемент в интерфейсе — обращение из кода остаётся по **предопределённому имени** метаданных.

**Пример:**
```1c
// Правильно: обращение к предопределенному элементу
КонтрагентДляДокумента = Справочники.Контрагенты.РозничныйПокупатель;

// Неправильно: обращение по имени
КонтрагентДляДокумента = Справочники.Контрагенты.НайтиПоНаименованию("Розничный покупатель");
```

---

#### 2.4 Иерархия справочников (5 минут)

Справочники могут быть иерархическими.

**Виды иерархии:**

1. **Иерархия групп** — элементы группируются в папки
   - Пример: номенклатура поделена на группы (запчасти, товары к отгрузке, услуги)
   - Группы служат для удобства пользователей

2. **Иерархия элементов** — сами элементы могут быть родителями и дочерними **внутри одного справочника**
   - Пример: юрлицо с обособленными подразделениями в других городах; у каждого подразделения — свой контрагент как дочерний элемент
   - У ИП обычно нет такой «ветки» — контрагент один, без дочерних элементов
   - Связь «контрагент → договоры» — это не иерархия элементов, а **подчинённый справочник** (см. п. 2.5)

**Визуализация** (один справочник **Контрагенты**):
```
Контрагенты
├── ООО "Рога и Копыта"                    (юрлицо — голова)
│   ├── Обособленное подразделение (Казань)
│   └── Обособленное подразделение (Новосибирск)
└── ИП Иванов                            (без дочерних — типично для ИП)
```

---

#### 2.5 Подчиненные справочники (5 минут)

**Подчиненные справочники** — справочники, элементы которых принадлежат элементам другого справочника.

**Примеры:**
- Договоры контрагентов (подчинены справочнику Контрагенты)
- Банковские счета контрагентов (подчинены справочнику Контрагенты и Физические лица)

**Настройка:**
- В свойствах справочника указываются владельцы (справочники-родители)
- При создании элемента владелец практически всегда присутствует
- Владелец можно не выводить на форму, если он всегда известен

**Важно:** Не путать с понятием "владелец" в ролевой модели доступа (это отдельная тема).

### Блок 3: Документы (25 минут)

#### 3.1 Назначение документов (5 минут)

**Документы** предназначены для отражения фактов хозяйственной, управленческой и другой деятельности предприятия.

**Отличия от справочников:**
- Справочники — каталожные данные
- Документы — фиксация событий и операций

**Пример:** Документ "Реализация товаров и услуг"
- Табличная часть "Товары" содержит номенклатуру, характеристики, цены, количество
- Склад может быть указан в шапке документа или в табличной части (зависит от того, с одного или нескольких складов реализация)

---

#### 3.2 Обязательные реквизиты документов (5 минут)

**Дата и номер** — обязательные реквизиты документа (нельзя отказаться).

**Отличия от справочников:**
- В справочнике можно отказаться от кода и наименования
- В документе дата и номер всегда присутствуют

**Предопределенные документы:**
- Обычно **нет** предопределённых экземпляров документов: документ отражает **конкретный факт** и получает номер/дату в жизни ИБ
- Исключения в отраслевых решениях возможны (на лекции упоминался кейс **Управление холдингом**: **договор как документ**, версии соглашений и влияние на графики платежей — при этом рядом может оставаться **справочник** договоров для «каталожного» слоя)
- **ГОСТ** в учётной модели чаще ближе к **справочнику** (каталожная ссылка с реквизитами номера/дат действия), а не к документу факта хозяйственной жизни организации-пользователя

**Пример структуры «оплата» (личные финансы / касса):**
- В шапке — **контрагент** и итоги; в **табличной части** — строки по **договорам** и **суммам**, если одним платежом закрывают несколько обязательств.

---

#### 3.3 Состояния документов и проведение (10 минут)

**Состояния документа:**

1. **Черновик** — документ создан и заполнен, но не проведен
2. **Проведен** — документ проведен, созданы движения в регистрах

**Проведение документа:**
- Проведение — это создание движений документа в регистрах учета
- По **стандартам разработки** и обычной архитектуре 1С отчёты опираются на **регистры**, а не на «прямой обход» всех документов — регистры проектируют под быстрые выборки
- Пример **черновика:** документ записан, но проведение запрещено, пока сумма по шапке не сойдётся с суммой строк ТЧ (контроль через обработку проведения / отказ)

**Типы регистров:**
1. Регистр накопления
2. Регистр бухгалтерии
3. Регистр расчета
4. Регистр сведений

**Журнал документов:**
- Показывает все движения, которые делает документ
- Пример: при реализации записываются движения в регистры:
  - Расчеты с клиентом
  - Себестоимость (может рассчитываться позже)
  - Товары к отгрузке
  - Товары организации

---

#### 3.4 Пометка на удаление (5 минут)

**Механизм удаления объектов:**

1. **Пометка на удаление** — пользователь помечает объект на удаление
2. **Процедура удаления помеченных объектов** — административная операция, которая:
   - Проверяет, что объект нигде не используется
   - Удаляет объект, если нет ссылок на него
   - Показывает ссылки, если объект используется

**Где в платформе (типовой путь):** **Администрирование → Обслуживание → Удаление помеченных объектов** — обработка есть даже в «пустой» конфигурации; предпочтительно запускать в **технологическое окно** / низкую нагрузку.

**Для документов:**
- При пометке на удаление все движения документа удаляются
- Проведенный документ с пометкой на удаление нельзя провести
- Можно снять пометку и провести документ заново

**Важность контроля:**
- В сложных системах (например, ERP) необходимо контролировать удаление
- Пользователи не могут напрямую удалять объекты (только помечать)
- Это предотвращает ситуации, когда есть проведенный документ с движением по несуществующему товару

**Пример сценария:**
1. Оприходован товар
2. Выяснилось, что товар неправильный
3. Помечаем на удаление документ и товар
4. Запускаем процедуру удаления помеченных объектов
5. Если товар нигде не используется — оба объекта удалятся
6. Если товар используется в других документах — удалится только документ, товар останется с указанием мест использования

### Блок 4: Журналы документов (5 минут)

#### 4.1 Назначение журналов

**Журналы документов** — группировка документов с общими реквизитами.

**Примеры:**
- Журнал "Поступления" объединяет:
  - Авансовый отчет (покупка сотрудником)
  - Приобретение товаров и услуг (обычная закупка)
  - Общий реквизит: контрагент-договор

- Журнал "Закупки" объединяет:
  - Заказы поставщика
  - Поступления
  - Счет-фактуры полученные

**Преимущества:**
- Все документы одного типа операций в одном месте
- Удобство для пользователей
- Единая навигация

---

### Блок 5: Перечисления (5 минут)

#### 5.1 Назначение перечислений

**Перечисления** — объект для хранения статической информации с небольшим количеством значений.

**Характеристики:**
- Нет реквизитов
- Нет табличных частей
- Пользователь может только выбирать из предопределенных значений
- Состав определяется разработчиком

**Примеры:**
- Виды товаров
- Пол (мужской, женский)
- Категории транспортных средств

**Когда использовать перечисление vs справочник:**

**Перечисление:**
- Очень короткий список значений
- Значения практически не меняются
- Нет дополнительных данных для каждого значения

**Справочник:**
- Значения могут изменяться
- Нужны дополнительные реквизиты
- Пример: ставки НДС (хотя относительно постоянны, но меняются, поэтому лучше справочник)

**Пример с НДС:**
- Можно было бы хранить в перечислении (18%, 10%, 20%)
- Но появились переходные ставки (6,66%)
- Лучше использовать справочник для гибкости

**Тезис лекции про историчность:** ставки **меняются** законодательно, а старые значения нужны для **исторических документов** — поэтому **ставки НДС** обычно **справочник** (часто с предопределёнными), а не перечисление; **вид контрагента** (юр/физ/ИП) при «замороженном» наборе — типичный кандидат в **перечисление**.

---

### Блок 6: Отчеты и обработки (3 минуты)

#### 6.1 Назначение

**Отчеты** — для построения отчетов и аналитики.

**Обработки** — для манипуляций с данными.

**Важно:** Это не объекты хранения!
- У них нет своих экземпляров
- Пользователь может формировать отчеты и выполнять обработки
- В них ничего не хранится

**Отличие от справочников и документов:**
- Справочники и документы — классы для хранения данных
- Отчеты и обработки — инструменты для работы с данными

---

### Блок 7: Планы видов характеристик (12 минут)

#### 7.1 Назначение и проблема (5 минут)

**Проблема:** Разные предприятия имеют разные требования к характеристикам объектов.

**Примеры:**
- Шинный завод: диаметр, ширина, индекс скорости
- Одежда: цвет, размер, пол, материал
- Невозможно заложить все варианты в универсальную конфигурацию

**Решение:** Планы видов характеристик позволяют пользователям настраивать характеристики в процессе эксплуатации.

**Пример:** Справочник "Номенклатура" с подчиненным справочником "Характеристики"
- Пользователь может добавить дополнительные реквизиты к характеристикам
- Например, "Ширина ленты" для конвейерной ленты

---

#### 7.2 Дополнительные реквизиты и сведения (7 минут)

**Два подхода:**

1. **Дополнительные сведения** (регистр сведений)
   - Хранятся в одном общем регистре сведений
   - Структура: объект, свойство, значение
   - **Проблема:** Объект слишком широкого типа — проблемы производительности при запросах

2. **Дополнительные реквизиты** (табличная часть)
   - Каждый справочник имеет табличную часть "Дополнительные реквизиты"
   - Структура: свойство, значение
   - **Преимущество:** Проще обращаться к данным, так как тип объекта конкретный

**Когда использовать:**

**Дополнительные сведения:**
- Для просмотра информации в самом элементе
- Когда не нужна агрегация данных

**Дополнительные реквизиты:**
- Когда нужна агрегация (отчеты, аналитика)
- Когда нужно строить запросы по этим данным

**Настройка:**
- Делается в пользовательском режиме (режиме настройки)
- Консультанты или продвинутые пользователи добавляют характеристики
- В плане видов характеристик определяются типы значений характеристик

**Пример на лекции (ERP «одна коробка — разные отрасли»):** шины — **размерность и сезонность**; одежда — **размер, цвет, пол** и т.д.; расширение без правок универсального ядра и упрощение обновлений.

**Доработки под предприятие:** вместо десятков констант для редких флагов (напр. **возвратные отходы** при расчёте себестоимости, разные категории по организациям) удобнее завести **ПВХ** и значения в пользовательском режиме.

---

### Блок 7а: План счетов, регистр бухгалтерии и регистр расчета (12 минут)

#### 7а.1 План счетов

- Используется там, где учёт строится по **счетам**: бухгалтерия, МСФО, бюджетирование.
- **Суть:** движение всегда «**откуда — куда**» — **дебет** и **кредит** с **разной аналитикой** на сторонах (пример с платежом: **банковский счёт** с одной стороны, **контрагент + договор + счёт на оплату** — с другой).

#### 7а.2 Основные средства и амортизация (учебная схема)

- Покупка компьютера не списывается целиком в месяц закупки (искажение прибыли и требования к **равномерному** учёту затрат/налогам): заводится **ОС**, первоначальная стоимость на условном счёте **01**, начисление **амортизации** частями на условный счёт затрат из остатка стоимости (на лекции — ориентир «100 тыс. / 10 месяцев по 10 тыс.»).
- Модернизация, дооценка, обесценение — смена **балансовой** стоимости во времени.

#### 7а.3 Регистр бухгалтерии

- По смыслу — **усложнение** идеи регистра накопления: **одна** хозяйственная операция порождает **корреспонденцию** сразу по **двум** «сторонам» счетов с разной аналитикой.

#### 7а.4 План видов расчета и регистр расчета

- В первую очередь **расчёт заработной платы**: оклад, премии, отпускные и др.
- **Виды расчёта** могут **вытеснять** друг друга по периоду (пример: сначала начислён оклад, затем период болезни вытесняет начисление за эти дни).
- В других предметных областях регистр расчёта встречается реже; при проектировании смотрят ТЗ и типовые паттерны.

### Блок 8: Регистры накопления (20 минут)

#### 8.1 Назначение регистров накопления (5 минут)

**Регистры накопления** используются для:
- Хранения количественных и суммовых данных
- Учета остатков и оборотов
- Построения отчетов

**Логика работы:**
- Документы (регистраторы) создают движения в регистрах; **независимых** регистров накопления **нет** — движения всегда от документа или иного регистратора платформы
- Движения бывают: приход (+) и расход (-)
- Система автоматически рассчитывает остатки и обороты

**Пример:** Регистр "Товары на складах"
- Измерения: номенклатура, характеристика, склад, помещение, серия
- Ресурс: количество

---

#### 8.2 Измерения и ресурсы (5 минут)

**Измерения** — разрезы, в которых хранятся данные.

**Пример для регистра "Товары на складах":**
- Номенклатура
- Характеристика (подчиненный справочник к номенклатуре)
- Склад
- Помещение
- Серия (для учета серийных номеров, например, по госконтракту)

**Ресурсы** — значения, которые накапливаются (количество, сумма).

**Важно:** По измерениям можно получать остатки и обороты.

**Ресурсы:** в регистре накопления ресурсы **суммируются и вычитаются** — это **числовые** величины; «сложить двух контрагентов» или две должности как ресурс нельзя. В **регистре сведений** в ресурсе (и измерении) допустимы **ссылочные** типы, хранилище значений и др.

---

#### 8.3 Виды регистров накопления (5 минут)

**1. Регистр остатков:**
- Остатки должны стремиться к нулю
- Пример: товары на складах (приход, затем расход)
- Приход и расход компенсируют друг друга

**2. Регистр оборотов:**
- Накопление **оборотов за периоды** без контроля «остатка на полке» в том же смысле, что у остаточного регистра
- Обычно **один вид движения** (нет пары приход/расход): показатели вроде **выручки** только **нарастают** по периодам
- Пример: регистр "Продажи"
- Используется для аналитики (продажи за месяц, неделю, день)
- Пример с лекции: **расход топлива** автопарка — удобнее смотреть **обороты по маршрутам/периодам**, чем «остаток топлива в баке» как учётную величину регистра остатков (каждый выезд — новая отправная точка)

**Пример использования:**
При проведении реализации товаров записываются движения в:
- Регистр остатков "Товары на складах" (контроль остатков)
- Регистр остатков "Товары организации" (внутрифирменный учет)
- Регистр оборотов "Продажи" (аналитика продаж)

---

#### 8.4 Реквизиты движений (5 минут)

**Реквизиты движений** — дополнительная информация о движении.

**Использование:**
- Для анализа: что это было за движение?
- Пример: корректировка реализации после закрытия месяца
- Помогает понять причину изменений в регистрах

**Пример сценария:**
1. Месяц закрыт
2. Выяснилось, что цена в договоре была другой
3. Создается корректировка реализации
4. Корректируется себестоимость товара
5. В реквизитах движения можно увидеть, что это была корректировка

---

### Блок 9: Регистры сведений (15 минут)

#### 9.1 Назначение регистров сведений (5 минут)

**Регистры сведений** используются для хранения информации в разрезе измерений.

**Отличия от регистров накопления:**
- Не накапливают остатки и обороты
- Хранят конкретные значения по измерениям

**Примеры:**
- Сотрудник + степень родства + родственник (для льгот)
- Контрагент + вид контактной информации + значение (телефон, адрес, email)

**Примеры с лекции:**

- **Цены номенклатуры** — нужна **история** во времени; константа для «текущей точки интеграции» не заменяет историю цен.
- **Должность сотрудника** во времени — периодический независимый РС: измерение **сотрудник** (и при необходимости **организация**), ресурс **должность** может быть **ссылкой**; для премий важно знать доли периодов «был стажёром / стал аналитиком».
- **Подчинение регистратору:** должность меняется только проведением **кадрового приказа** — прямой ручной ввод в РС недоступен.
- **Реквизиты движения** РС — служебная информация среза (напр. **алгоритм расчёта цены** — ссылка на справочник); в отчётах по «срезу» может быть недоступно, если отчёт строится не по полной таблице движений.
- **Ключ учёта взаиморасчётов** — справочник «сжимает» связку контрагент–договор–счёт в одно измерение в регистрах; **расшифровка** состава ключа хранится в **независимом** регистре сведений (экономия ширины и связей в движениях).
- **Штрихкод → номенклатура + характеристика:** РС с измерением **штрихкод** (строка) и ресурсами **номенклатура** и **характеристика**; в рамках одной ИБ подразумевается **однозначное** соответствие (ввод через регистратор штрихкодов; коллизии при «ручном дублировании» возможны как ошибка ввода).

---

#### 9.2 Периодические регистры сведений (5 минут)

**Периодические регистры** — регистры с периодичностью (день, месяц и т.д.).

**Пример:** Курсы валют
- Измерение: валюта
- Периодичность: день
- Значение: курс валюты

**Особенности:**
- В один период можно записать только одну запись с одинаковыми измерениями
- Система не даст записать дубликат в тот же день
- Если курс не меняется несколько дней — каждый день все равно создается запись

**Использование:**
- Срез последних — получить актуальный курс (даже если запрашиваешь в выходной)
- Срез первых — получить ближайшее изменение курса (например, курс на понедельник при запросе в воскресенье)

---

#### 9.3 Подчиненные и независимые регистры (5 минут)

**Подчиненные документу:**
- Запись может сделать только документ
- Пример: должности сотрудника организации
- Кадровик не может напрямую изменить должность
- Изменение происходит через документ "Прием на работу" или "Кадровое перемещение"

**Независимые:**
- Можно изменять напрямую
- Пример: дополнительные сведения об объектах
- Если свойства нет — добавляем
- Если есть — изменяем существующую запись

**Важно:** Система не даст создать вторую запись с такими же измерениями.

---

### Блок 10: Регистры бухгалтерии (5 минут)

#### 10.1 Назначение и особенности

**Регистр бухгалтерии** используется для ведения бухгалтерского учета с корреспонденцией счетов.

**Особенности:**
- Один регистр для всех операций
- Представляет собой набор маленьких регистров накопления, которые корреспондируют между собой
- Важен для бухгалтерской отчетности

**Производительность:**
- При записи и построении отчетов могут возникнуть проблемы производительности
- В системах с высокой нагрузкой (ERP, УТ) регистр бухгалтерии заполняется позже
- Основной учет ведется через регистры накопления (быстрее)

**Примеры систем:**
- Бухгалтерия-предприятие: регистр бухгалтерии — основной объект
- УТ, ERP, КА: основой является управленческий учет через регистры накопления

---

### Блок 11: Итоги (5 минут)

#### 11.1 Назначение итогов

**Итоги** — предварительно рассчитанные данные для ускорения работы системы.

**Зачем нужны:**
- В системе может быть огромное количество документов и движений
- Для быстрого получения данных система хранит итоги

**Пример:** Остатки товаров
- При поступлении товара система обновляет итоги
- При запросе остатков система обращается к итогам (быстро, из кэша)
- При запросе остатков на конкретную дату система использует итоги, рассчитанные на определенные периоды

**Управление итогами:**
- Пользователь косвенно влияет на итоги через движения
- Физически существует таблица итогов
- Можно задать период для итогов (для получения данных на конец предыдущего года)

## Резюме лекции (5 минут)

### Ключевые моменты:

1. **Константы** — реквизиты конфигурации для условно постоянных величин
   - Можно хранить ссылочные типы и хранилища значений
   - Изменение может повлиять на логику системы

2. **Справочники** — для каталожных данных
   - Код и наименование (можно отказаться)
   - Табличные части для множественных записей
   - Предопределенные элементы для обращения в коде
   - Иерархия групп и элементов
   - Подчиненные справочники

3. **Документы** — для отражения фактов деятельности
   - Обязательны дата и номер
   - Состояния: черновик, проведен
   - Проведение создает движения в регистрах
   - Пометка на удаление с контролем использования

4. **Журналы документов** — группировка документов с общими реквизитами

5. **Перечисления** — статические короткие списки значений
   - Нет реквизитов и табличных частей
   - Пользователь только выбирает

6. **Планы видов характеристик** — настройка характеристик пользователем
   - Дополнительные сведения (регистр) vs дополнительные реквизиты (табличная часть)
   - Выбор зависит от необходимости агрегации

7. **Регистры накопления** — для учета остатков и оборотов
   - Измерения (разрезы учета)
   - Ресурсы (накапливаемые значения)
   - Регистры остатков и оборотов
   - Реквизиты движений для анализа

8. **Регистры сведений** — хранение значений по измерениям
   - Периодические (курсы валют)
   - Подчиненные документу vs независимые
   - Срез последних/первых для периодических

9. **Регистры бухгалтерии** — для бухгалтерского учета
   - Корреспонденция счетов
   - Могут быть проблемы производительности
   - В системах с высокой нагрузкой заполняются позже

10. **Итоги** — предрассчитанные данные для производительности
    - Обновляются при движениях
    - Используются для быстрого получения остатков и оборотов

11. **План счетов / регистр бухгалтерии** — корреспонденция и аналитика по счетам; учебный пример **ОС и амортизации**

12. **План видов расчета / регистр расчета** — периодические расчёты (ЗП), **вытеснение** видов расчёта

---

### Практические рекомендации:

1. **Выбор объекта хранения:**
   - Константы — для настроек системы
   - Справочники — для справочных данных
   - Документы — для фактов деятельности
   - Регистры — для аналитики и учета

2. **Производительность:**
   - Используйте регистры накопления для управленческого учета
   - Регистр бухгалтерии заполняйте позже для высоконагруженных систем
   - Используйте итоги для ускорения работы

3. **Контроль данных:**
   - Используйте предопределенные элементы вместо обращения по имени
   - Применяйте пометку на удаление с контролем использования
   - Для периодических данных используйте периодические регистры сведений

4. **Гибкость:**
   - Планы видов характеристик для настраиваемых характеристик
   - Дополнительные реквизиты для агрегации данных
   - Дополнительные сведения для простого просмотра
