# Семинар 3: Длительные операции (многопоточность и отладка) и введение в Vanessa Automation (BDD/Gherkin)

**Продолжительность:** ~75 минут  
**Целевая аудитория:** Студенты 3 курса, прошедшие курс разработки на платформе 1С:Предприятие (1–2 семестры)  
**Курс:** Библиотека стандартных подсистем  
**Источник занятия:** транскрибация «БСП и Vanessa» (2026-03-13); демо на **Vanessa Automation** и учебной конфигурации с документом **«Реализация товаров и услуг»**

## Цель семинара

Завершить практику по **длительным операциям БСП**: обработка результата **нескольких потоков** (структура **ключ → значения**), настройка **режима отладки**, чтобы не отлаживать код в «шумной» среде с множеством чужих фоновых заданий. Познакомиться с **Vanessa Automation** как с **open source**-фреймворком для **автоматизации сценариев** в стиле **Gherkin** (BDD): отдельная **лёгкая** информационная база **менеджера тестирования**, подключение **клиента тестирования**, запуск с ключом **`/TestManager`**, запись и проигрывание простого сценария проверки **наличия команды** «Заполнить табличную часть» на форме документа — как задел под **регрессию** и **CI/CD**.

## План семинара (~75 минут)

> **Примечание:** Время сильно зависит от скорости скачивания дистрибутива Vanessa Automation, сетевых условий и повторения шагов за преподавателем.

### Блок 1: Многопоточное выполнение и приём результата (10 минут)

#### 1.1 Что поменялось в коде заполнения

- Логика получения данных для строк табличной части переводится на работу со **структурой ключ–значение** (или соответствием), которую возвращает/обрабатывает серверный код.
- Вместо одного вызова используется API **`ВыполнитьФункциюВНесколькоПотоках`** (аналогично по идее — **`ВыполнитьПроцедуруВНесколькоПотоках`**): для каждого «потока» задаётся **свой набор параметров**.
- На клиенте в обработчике завершения нужно **обойти** результаты по **ключам**, для каждого ключа забрать данные из **временного хранилища** (по адресам из описания результата), обработать ошибки и **собрать** итоговую табличную часть (например, методами дополнения таблиц из БСП — по аналогии с разбором одного результата).

#### 1.2 Зачем так делить работу

- Ускорение за счёт **параллельного** выполнения на сервере при корректной декомпозиции задачи.
- Важно не путать **ключи** параметров и **ключи** результатов при разборе соответствия.

### Блок 2: Отладка длительных операций без «фона» (8 минут)

#### 2.1 Проблема

- В **крупной** конфигурации одновременно выполняется много **фоновых заданий** — при отладке «в фоне» легко **упереться** в чужие задания, усложняется воспроизведение шага.

#### 2.2 Режим отладки (параметры запуска / сервисные настройки БСП)

- В **параметрах запуска** (контекст занятия — сервис поддержки/настройки БСП) включается **режим отладки** длительных операций.
- В этом режиме выполнение идёт **в том же потоке**, что и интерактивное приложение: **настоящей многопоточности** в смысле фоновых заданий нет, зато **стабильно** отрабатывают точки останова в коде менеджера и связанных модулях.
- После отладки режим **отключают**, чтобы снова проверить поведение «как в бою».

### Блок 3: Зачем автотесты и что такое Vanessa (12 минут)

#### 3.1 Конфликты в команде

- На проектах с несколькими разработчиками и консультантами типична ситуация: правки одного **ломают** сценарий другого (роли, интерфейс, доступность команд).
- Без автоматической проверки это превращается в споры и риск пронести регрессию в релиз.

#### 3.2 Vanessa Automation

- **Vanessa Automation** — распространяемый **open source** фреймворк для автоматизации тестирования **1С** (в т.ч. сценарии на **Gherkin** / **BDD**).
- Исходники и развитие — в **Git**; к проекту можно подключаться как участник.
- Не путать с тяжёлой **промышленной** конфигурацией: **менеджер тестирования** должен быть **максимально лёгким**.

#### 3.3 Почему не тестировать «из той же ERP»

- Использовать большую конфигурацию как **менеджер тестирования** — **плохая** практика: лишний вес, лишние объекты и фоновая активность.
- Целевой подход: отдельная **минимальная** база только под запуск менеджера.

### Блок 4: Минимальная ИБ менеджера тестирования (10 минут)

#### 4.1 Состав конфигурации

- Практически **пустая** конфигурация: без прикладных объектов курса.
- Минимум для работы: роль **«Полные права»** (иначе нельзя нормально создать пользователя для запуска клиента).
- При необходимости — один пользователь (чтобы не вводить пароль при каждом старте в учебной среде).

#### 4.2 Локальная файловая база

- Для учебного демо удобна **локальная** файловая база менеджера: проще обеспечить совместимость и воспроизводимость без донастройки сервера (на проде требования могут отличаться).

#### 4.3 Версия платформы и режим совместимости

- На занятии использовалась связка с **Vanessa Automation 3.0.7** (условное обозначение **VA 307** в материалах) и платформа **8.5**; для успешного открытия обработки может понадобиться задать **режим совместимости** конфигурации (например, **8.3.27** — уточнять по документации к **вашей** сборке VA и платформы).

#### 4.4 Интерфейс

- В свойствах конфигурации включить отображение в режиме **«Такси»** (и при необходимости снять избыточные ограничения интерфейса для демонстрации).

### Блок 5: Подготовка Vanessa Automation в менеджере (15 минут)

#### 5.1 Форматы поставки

- **Single** — один файл **epf/cfe** со всем содержимым (удобно для старта).
- **Набор файлов** — «разобранный» вариант с отдельными обработками-библиотеками.

#### 5.2 Администрирование безопасности

- В базе менеджера в **администрировании**: снять блокирующие опции (**защита от опасных действий**), чтобы была возможность подключать **внешние компоненты** / открывать **Vanessa Automation**.

#### 5.3 Обязательные флаги в Vanessa

- **Использовать внешнюю компоненту** (требование инструмента).
- Включить использование **Gherkin** (язык сценариев **Given–When–Then** и русскоязычные шаги в обвязке VA).
- Настроить **системный каталог** (каталог на диске), где лежат **feature**-файлы и связанные ресурсы сценариев (на занятии — папка рядом с учебной базой, условно **«307»** / каталог инструментов).
- Опционально: включить вычисление выражений **в фигурных скобках** в текстах шагов (настройка интерфейса VA).
- **Сохранить настройки**; при необходимости **выгрузить** их в файл — чтобы версионировать в **Git** и видеть, кто что менял.

### Блок 6: Клиент тестирования и запуск менеджера (12 минут)

#### 6.1 Добавление клиента

- В форме Vanessa добавить **клиент тестирования**: строка с параметрами подключения к **тестируемой** базе (учебная БСП с документом **«Реализация…»**).
- Тип клиента — **тонкий**; строка подключения — как для обычного запуска предприятия (скопировать из ярлыка / списка баз).

#### 6.2 Имя клиента

- Задать **осмысленное имя** клиента (на занятии — по шаблону имени учебной базы, например с суффиксом **`_307`**).
- Имя и **синоним** (если используется) лучше заполнять **согласованно**: часть шагов библиотек может опираться на одно из представлений.

#### 6.3 Режим Test Manager

- Менеджер тестирования нужно запускать **с ключом** командной строки, указывающим режим **менеджера тестирования** (на занятии — в **дополнительных параметрах** запуска указан вариант вида **`/TestManager`** — уточнить точное написание в справке **вашей** версии платформы и VA).

#### 6.4 Первое подключение

- Сценарий **«подключить / открыть клиент тестирования»** (шаги из библиотеки VA): проверка, что тестовый клиент стартует и подключается к нужной ИБ; при ошибках сверять имя клиента, строку подключения и режим запуска менеджера.

### Блок 7: Запись сценария и проверка команды на форме (18 минут)

#### 7.1 Зачем Gherkin в CI/CD

- Сценарии можно хранить как текст и запускать в **конвейере**: если сборка «зелёная», релиз-менеджеру не нужно вручную прогонять одни и те же дымовые сценарии.

#### 7.2 Постановка учебной проверки

- **Фича/сценарий:** на форме документа **«Реализация товаров и услуг»** доступна команда **«Заполнить табличную часть»** (или эквивалент по метаданным — уточнять в конфигурации).
- Это **защита от регрессии прав:** если коллега добавит роль и **отберёт** видимость команды у тестового пользователя, автопрогон **упадёт** до попадания в релиз.

#### 7.3 Запись поведения

- Подключить клиент тестирования по текущей строке настроек.
- Включить **запись поведения** в Vanessa; в тестируемой базе: открыть подсистему курса, создать документ, нажать кнопку заполнения табличной части.
- Остановить запись: в редакторе появятся шаги на **языке шагов** (библиотека командного интерфейса, ожидание открытия форм, нажатие кнопки по **внутреннему имени** команды).

#### 7.4 Прогон

- Сохранить сценарий и выполнить: при успехе шаги отображаются как **пройденные** — значит, цепочка UI-действий **воспроизводится**; при смене прав/ролей сценарий сломается осознанно.

## Резюме семинара (5 минут)

### Ключевые моменты

1. **Многопоточные длительные операции** БСП: свои **наборы параметров** на поток; результат — **набор адресов/значений по ключам**; клиентский код должен **собрать** итог без дублирования логики «в обход» платформы.
2. **Отладка:** режим, выполняющий код **в том же потоке**, помогает не тонуть в чужих фоновых заданиях; для приёмочных проверок режим отключают.
3. **Vanessa Automation** — **BDD/Gherkin**-ориентированный **open source** инструмент; менеджер тестирования — в **лёгкой** отдельной ИБ.
4. В менеджере: **полные права**, снятие **опасных действий**, **внешняя компонента**, **каталог сценариев**, при необходимости — **выгрузка настроек** в Git.
5. **Test Manager** запускается с **соответствующим ключом** платформы; к тестируемой базе подключается **клиент тестирования**.
6. **Запись сценария** фиксирует **реальные** шаги UI; простой тест на **наличие команды** уже ловит ошибки **ролевой модели**.

---

### Практические рекомендации

1. Хранить **feature**-файлы и **настройки VA** рядом с репозиторием прикладного решения; не коммитить пароли — использовать параметры среды/секреты CI.
2. Имена **внутренних** команд и форм брать из метаданных / записанного сценария, не «с экрана», если шаги ориентируются на **имена**.
3. После обновления **платформы или VA** сверять **режим совместимости** и список поддерживаемых режимов интерфейса (в т.ч. **8.5** и **такси**).
4. Для длительных операций: отладка в «однопоточном» режиме — только этап разработки; финальная проверка — **с фоновыми заданиями**.
