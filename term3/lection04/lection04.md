# Лекция 4: Подсистема «Обновление информационной базы» (БСП)

**Продолжительность:** ~45–50 минут  
**Целевая аудитория:** Студенты 3 курса, прошедшие курс разработки на платформе 1С:Предприятие (1–2 семестры)  
**Курс:** Библиотека стандартных подсистем  
**Источник занятия:** материал «БСП 06. Обновление ИБ» (транскрибация практической демонстрации на демоконфигурации БСП 3.1.x)

## Цель лекции

Разобрать **подсистему обновления версии информационной базы** в БСП: зачем нужны **обработчики обновления**, как связаны **версии подсистем** и **версии конфигурации**, как зарегистрировать **собственную библиотеку** и **обработчики**, как безопасно **переносить данные** при изменении структуры (пример: склад из шапки документа в табличную часть), и чем отличаются режимы выполнения: **монопольный**, **оперативный**, **отложенный**.

## Необходимые предзнания

- Состав БСП, публичный и переопределяемый API (лекции 1–2)
- Понятие **версии конфигурации** (редакция, подредакция, сборка)
- Работа с **общими модулями**, **модулями менеджера** документа, **запросами**
- Навигация по инструкции БСП на **ИТС** (раздел по разработке, версия библиотеки)

## План лекции (~45–50 минут)

> **Примечание:** на занятии используется инструкция с портала разработки (в транскрипте — ссылка на ресурс с материалами по БСП; в рабочей практике ориентируйтесь на **ИТС** и версию **БСП 3.1.x / 3.2.x** вашей поставки).

### Блок 1: Зачем нужна подсистема обновления ИБ (8–10 минут)

#### 1.1 Типовая бизнес-ситуация

**Пример из демонстрации:**

- Есть документ **«Реализация товаров и услуг»**.
- Раньше **склад** был в **шапке** документа.
- Требования изменились: склад нужно указывать в **табличной части** (по строкам).
- Разработчик изменил метаданные и код проведения так, что движения формируются **из строк табличной части**.

**Проблема после простого «обновления конфигурации»:**

- Новый реквизит табличной части **пустой**.
- Старый реквизит шапки (например, помеченный как **удаляемый**) может **остаться заполненным** до удаления реквизита на следующем этапе.
- **Стандартными средствами** платформы одной кнопкой «обновить» **не перенести** автоматически значение из шапки в строки — нужен **прикладной** код именно в момент смены версии.

#### 1.2 Роль подсистемы БСП

**Подсистема обновления версии ИБ:**

- Хранит/отслеживает, **какая версия** была у конфигурации (и подсистем).
- При смене версии **запускает цепочку обработчиков** — для каждого интервала «было → стало».
- Если, например, база была на версии **2.0**, а обновляетесь на **3.0**, выполняются обработчики для всех промежуточных правил между этими версиями (в соответствии с настройками и документацией БСП).

**Итог:** обновление — это не только загрузка новой структуры метаданных, но и **прикладная миграция данных** под новую логику.

### Блок 2: Версии подсистем и «основная конфигурация» (6–8 минут)

#### 2.1 Регистр сведений «Версии подсистем»

Через **функции для технического специалиста** открывают регистр (в типовой терминологии БСП — **версии подсистем** / аналог по вашей редакции).

**В регистре видно:**

- какие **библиотеки** (подсистемы) подключены и их **версии**;
- у одной из строк может быть признак **основной конфигурации** (в демо: совпадение имени библиотеки с именем конфигурации, например «Библиотека стандартных подсистем демо»).

#### 2.2 Почему важен параметр запуска «обновить ИБ»

Если библиотека **не** помечена как основная конфигурация, **автоматический** запуск полного цикла обновления при смене версии **может не произойти** так, как ожидает разработчик.

**Практический вывод:** для отработки обработчиков и актуализации регистра версий нужно уметь запускать ИБ с параметром **обновления информационной базы** (в транскрипте — «Запустить обновление информационной базы»; точная формулировка ключа — по документации к вашей версии платформы и БСП).

Без этого **запись в регистре** может **не обновиться**, и **обработчики не вызовутся**, даже если версию в коде изменили.

### Блок 3: Своя «библиотека» и модуль обновления (12–15 минут)

#### 3.1 Шаблон общего модуля

**Типовой путь (как на занятии):**

1. Найти общий модуль вида **`ОбновлениеИнформационнойБазыБСП`** (имя зависит от поставки).
2. **Скопировать** модуль.
3. Убрать суффикс **`БСП`** и задать **постфикс**, соответствующий **сокращению вашей библиотеки** (в примере: общий модуль общего назначения **«Мой проект»** → модуль **`ОбновлениеИнформационнойБазыМП`**).
4. Открыть модуль и **очистить реализации** процедур/функций от лишнего кода демо (оставить каркас, куда вы вставите свою логику).

**Замечание из лекции:** часть методов в шаблоне нужна только если ваша библиотека является **конечной конфигурацией**; в учебном сценарии их можно временно оставить.

#### 3.2 Процедура «При добавлении подсистемы»

Заполнить **описание**:

- **`Имя`** — имя вашей библиотеки (например, **«Мой проект»**).
- **`Версия`** — **строка** в формате БСП: **редакция**, **подредакция**, **версия**, **сборка** (семантика приращений — как на курсе по конфигурациям: правка ошибок vs новый функционал vs изменение логики vs крупный релиз).

**Зависимости:**

- **`Требуемые подсистемы`** — например, указать, что перед вашей подсистемой должны обновиться **стандартные подсистемы** (БСП).

#### 3.3 Подключение из переопределяемого модуля подсистемы конфигурации

**Модуль:** `ПодсистемыКонфигурацииПереопределяемый` (или актуальное имя в вашей поставке).

**Процедура:** **`ПриДобавленииПодсистем`**:

- параметр — массив модулей подсистем;
- в массив добавить **имя серверного модуля** вашего обновления, например **`ОбновлениеИнформационнойБазыМП`**.

**Рекомендация БСП:** в переопределяемом модуле **не размещать** объёмную логику — только **вызов** одноимённого метода из вашего общего модуля (`ОбновлениеИнформационнойБазыМП.ПриДобавленииПодсистем` и т.д.).

#### 3.4 Регистрация модуля в составе подсистем

В **составе подсистем** конфигурации нужно добавить **имя** нового общего модуля обновления.

#### 3.5 Проверка после запуска с параметром обновления

После запуска с параметром обновления ИБ в регистре **«Версии подсистем»** должна появиться строка **вашей** библиотеки с указанной **версией**.

---

### Блок 4: Обработчики обновления (10–12 минут)

#### 4.1 Процедура «При добавлении обработчиков обновления»

В общем модуле вашей библиотеки — метод **`ПриДобавленииОбработчиковОбновления`** (в документации БСП — с подробным **примером** в комментарии к типовому модулю).

**Обработчики** задаются через **таблицу значений** с колонками, описанными в БСП (см. также структуру **`НоваяТаблицаОбработчиковОбновления`** / аналог в вашей версии).

#### 4.2 Колонка «Версия»

- Версия задаёт **точку**, при переходе через которую (или на которую) должен выполниться обработчик.
- **Пример из речи:** если в регистре была **1.0.1.1**, а вы обновляетесь на **1.0.2.2**, может подойти обработчик с версией **1.0.2.1** — он попадает в «окно» между старой записанной версией и целевой.

#### 4.3 Указание процедуры-обработчика

В обработчике указывается **метод**, который будет вызван. В демо это экспортная процедура в **модуле менеджера** документа **«Реализация товаров и услуг»**, например:

**`ЗаполнитьСкладВТабличнойЧастиПриОбновлении`**.

#### 4.4 Режим выполнения (вводно)

В таблице обработчиков задаётся **режим выполнения**:

- **монопольно**;
- **оперативно**;
- **отложено**.

На первом проходе в демо выбран режим **монопольный**; детальный разбор — в конце лекции.

#### 4.5 Обработчики «на любую версию»

Для обработчиков, которые должны выполняться **при каждом** обновлении подсистемы **независимо** от номера версии, в БСП допускается указание версии **«*»** (звёздочка). *(Уточняйте поведение по инструкции вашей редакции БСП.)*

### Блок 5: Пример кода обработчика (перенос склада) (12–15 минут)

#### 5.1 Алгоритм в модуле менеджера документа

**Задача:** для уже существующих документов:

- в **табличной части** «Товары» склад **не заполнен** (пустая ссылка);
- при этом в **удаляемом** реквизите шапки **«УдалитьСклад»** значение **есть**.

**Запрос:**

- выбрать **уникальные ссылки** на документы, удовлетворяющие условию (в конструкторе — опция **«Без повторяющихся»** при обращении к табличной части);
- условия по строкам ТЧ и по шапке — по смыслу задачи.

#### 5.2 Обход и запись

Для каждой ссылки:

- получить **объект документа**;
- для строк табличной части, где склад пустой, проставить склад из **`ДокументОбъект.УдалитьСклад`** (или актуальное имя удаляемого реквизита);
- **записать** документ.

#### 5.3 Отключение побочных эффектов при записи

Так как **логика проведения** по смыслу **не меняется** (данные лишь «раскладываются» по новой структуре), важно **не вызвать** лишние обработчики, обмен, интеграции:

- **`ОбменДанными.Загрузка = Истина`** — типовой приём подавления части прикладной логики при программной записи;
- **`ДокументОбъект.ОбменДанными.Получатели.АвтоЗаполнение = Ложь`** — чтобы объект **не зарегистрировался** автоматически на планах обмена с авторегистрацией.

**Упрощение:** в модуле обновления БСП есть готовый метод **`ЗаписатьОбъект`** (в транскрипте — «записать объект» из API подсистемы) — он **инкапсулирует** типовые настройки обмена для сценария обновления. В демо передают **документ-объект** в этот метод.

### Блок 6: Отладка и повторный запуск обработчика (5–7 минут)

#### 6.1 Тестовые данные

В пользовательском режиме создать ситуацию: в строках **нет** склада, в шапке (удаляемый реквизит) склад **есть**; при необходимости **не проводить** документ, если проверка заполнения мешает (в демо всплыла ошибка в обработчике проверки заполнения — её исправили: **корректно обработать пустую ссылку на склад** перед вызовами функций, завязанных на «использование складских помещений» и т.п.).

#### 6.2 Отладка фоновых сценариев

Для отладки обновления в фоне — добавить параметр запуска **режима отладки** (как в курсе по платформе — см. документацию к версии 1С).

#### 6.3 Повторный запуск того же обработчика

**Вариант 1:** в регистре **«Версии подсистем»** **понизить** версию своей библиотеки (включить редактирование записи, ввести, например, **1.0.1** вместо более новой) — при следующем обновлении цепочка может выполниться **снова**.

**Вариант 2:** временно в **`ПриДобавленииОбработчиковОбновления`** подставить версию вроде **«ОтладкаОбработчика»** / ослабить условие — только для **dev**; в бою так не оставлять.

### Блок 7: Режимы выполнения: монопольный, оперативный, отложенный (6–8 минут)

*(По документации БСП и ремаркам лектора.)*

#### 7.1 Монопольный режим

- Нужен, когда данные **должны быть готовы до входа пользователей**.
- Примеры: критичные справочники, **активные** позиции, небольшой объём **текущего периода** — то, без чего работа в системе **невозможна** или опасна.

#### 7.2 Оперативный режим

- Чаще для **исправительных релизов**, в т.ч. при **динамическом обновлении**.
- **Важно:** если есть хотя бы один **монопольный** обработчик обновления конфигурации, **оперативные** обработчики **выполняются в монопольном контексте** (формулировка из лекции — сверять с редакцией ИТС).
- Если обработчик действительно выполняется при **активных сеансах** и регламентных заданиях — необходимо **программно обеспечивать блокировки** данных на прикладном уровне.

#### 7.3 Отложенный режим

- Запускается **после** монопольных и оперативных.
- Предназначен для **больших объёмов**; может выполняться **многопоточно** (несколько фоновых заданий) — **ускоряет** обработку.
- В типовых конфигурациях БСП **наиболее часто** используется именно **отложенная** обработка для «тяжёлых» миграций.

## Резюме лекции

### Ключевые моменты

1. При изменении структуры и логики (пример — перенос склада из шапки в ТЧ) **одного обновления КД недостаточно**: нужны **обработчики обновления**, которые перенесут данные в момент смены версии.
2. Подсистема БСП отслеживает **версии** и запускает обработчики по правилам **диапазонов версий**.
3. **«Версии подсистем»** — ключевой служебный регистр; признак **основной конфигурации** влияет на автоматизм запуска; часто нужен **явный** запуск с параметром **обновления ИБ**.
4. Своя библиотека: копия и адаптация **`ОбновлениеИнформационнойБазы…`**, заполнение **`ПриДобавленииПодсистем`**, зависимость от **БСП**, вызов из **`ПодсистемаКонфигурацииПереопределяемый`**.
5. Обработчики регистрируются в **`ПриДобавленииОбработчиковОбновления`**; версия определяет **момент** запуска; есть режим **«*»** для повторяемых сценариев (по документации).
6. При записи из обработчика используют **`ОбменДанными`**, **`ЗаписатьОбъект`** API БСП — чтобы не сломать обмен и не дублировать регистрацию.
7. **Монопольный** — критично к старту; **оперативный** — нюансы с монополией и блокировками; **отложенный** — большие объёмы и многопоточность.

### Вопросы для самопроверки

1. Почему после переноса реквизита из шапки в табличную часть «просто обновить конфигурацию» недостаточно для корректных данных?
2. Как подсистема БСП понимает, **какие** обработчики запустить при переходе с версии A на версию B?
3. Зачем нужен регистр **«Версии подсистем»** и что означает признак **основной конфигурации**?
4. Какие шаги нужны, чтобы **подключить** собственный модуль обновления к БСП?
5. Что такое **`ПриДобавленииПодсистем`** и **`ПриДобавленииОбработчиковОбновления`** и чем они отличаются по назначению?
6. Как в демо выбиралась **версия** обработчика относительно версии, записанной в регистре?
7. Зачем при записи документа из обработчика выставляют **`ОбменДанными.Загрузка`** и отключают автозаполнение получателей обмена?
8. Зачем использовать **`ЗаписатьОбъект`** из подсистемы обновления БСП?
9. Как **повторно** отладить обработчик без выпуска новой версии продукта?
10. Чем **монопольный**, **оперативный** и **отложенный** режимы отличаются по назначению и рискам?
11. Почему отложенные обработчики часто используют для больших объёмов данных в типовых решениях?

### Практические шаги после лекции

1. На ИТС открыть раздел **БСП** вашей редакции и найти главу про **обновление версии ИБ** и таблицу обработчиков.
2. В учебной демобазе повторить цепочку: **модуль обновления** → **переопределяемый** → **запуск с обновлением ИБ** → проверка регистра версий.
3. Реализовать **учебный** обработчик (упрощённо) и пройти цикл **понижения версии** в dev для повторной отладки.
