# Лекция 4: Динамические списки и параметры сеанса

**Продолжительность:** ~90–120 минут  
**Целевая аудитория:** Студенты 2 курса, Факультет информационных технологий  
**Курс:** Разработка на платформе 1С:Предприятие  
**Источник занятия:** `Лекция_04_Динамические_списки_Параметры_сеанса_2026-03-21.txt`

## Цель лекции

Закрепить различие **выгрузки конфигурации** (**.cf**) и **полной выгрузки ИБ** (**.dt**), роль **бэкапа СУБД**. Дать обзор **проектных подходов**: реактивная поддержка без методологии, **PMBOK / «водопад»**, **Agile**-манифест, **Scrum** и **Kanban**. На практике разобрать **динамический список** в форме **выбора номенклатуры** с **остатками** по складу: произвольный запрос, **основная таблица** справочника vs ошибка «только с остатком», **левое соединение**, галочки конструктора, **динамическое считывание данных**, назначение **формы выбора** в табличной части **«Расход товара»**, **параметры выбора** и **параметры запроса** к **виртуальной таблице** остатков регистра накопления, поведение при **непереданном** складе. Реализовать **параметры сеанса** «**текущий пользователь**» через справочник **ссылочного** типа, **модуль сеанса**, предопределённый элемент **«не авторизован»**, создание элемента при первом входе; связать с **ролями** и **журналом регистрации** при отсутствии прав на справочник пользователей.

## Необходимые предзнания

- Лекция 3 / семинар 3: пользователи, роли, запуск под другим пользователем (**Сервис → Параметры → Запуск**)
- Справочники, документы, регистр накопления, проведение
- Базовые запросы и конструктор запроса

## План лекции (~90–120 минут)

> **Примечание:** Начало занятия — викторина в Telegram; напоминание про выгрузку **.cf** преподавателя для подхвата конфигурации.

### Блок 0: Напоминание: .cf, .dt и СУБД (5–8 минут)

- **CF** (в речи — «cf-ник») — выгрузка **метаданных** без данных; такой файл можно получать **без выгона пользователей**.
- **Выгрузка ИБ из конфигуратора** «метаданные + данные» — обычно нужно, чтобы **не было сеансов** предприятия; платформа **сама** не обеспечивает консистентность данных при работающих пользователях так же, как при «холодной» логике выгрузки.
- **Бэкап средствами СУБД** — можно делать **на горячую**; СУБД обеспечивает **консистентность** снимка (см. также лекцию про **pg_dump**).

### Блок 1: Проектные технологии (18–22 минуты)

#### 1.1 «Делаем — исправляем» (реактивная поддержка)

- Единственный разработчик в организации, нет формальной методологии: правки **по запросам**, обновления типовых, исправление ошибок — **бесконечный цикл**.
- **Нельзя** прогнозировать сроки и бюджет внедрения; качество системы со временем **деградирует** без осознанного процесса.

#### 1.2 PMBOK и «водопад»

- Подходит для **повторяющихся**, **предсказуемых** проектов (пример с записи — **строительство** по типовому плану).
- Этапы идут **последовательно** (как «водопад» на диаграмме: одна работа закончилась — следующая началась).
- На **длинных** ИТ-проектах требования **меняются** — жёсткий план даёт **неточность** сроков и результата.

#### 1.3 Agile-манифест (2001) и Scrum

- Ценности: **люди и взаимодействие**, **работающий продукт**, **сотрудничество с заказчиком**, **готовность к изменениям** — важнее формальных артефактов и первоначального плана.
- **Scrum**: общий план продукта, **короткие спринты**, приоритеты с **заказчиком**, оценка сложности, **ретроспектива**, непрерывное улучшение — уместно, когда **конечный продукт** заранее не полностью ясен.

#### 1.4 Kanban

- Удобен для **поддержки**: доска **этапов** (например, ТЗ → разработка → тест → сдача); видно **скопление** задач на узком участке → добавить ресурсы или менять процесс.

### Блок 2: Динамические списки: форма выбора с остатками (35–45 минут)

#### 2.1 Задача

- В **расходе товара** при выборе номенклатуры нужно видеть **количество на складе**; в **приходе** остаток в форме выбора **не обязателен** (фиксируется факт поступления).

#### 2.2 Формы выбора

- Добавляют **форму выбора** справочника **«Номенклатура»**; колонку **код** можно **скрыть**; хороший тон — добавить **невидимую ссылку** на элемент для последующего расширения полями.
- Копируют форму выбора в вариант **«с количеством»**; **основной** формой выбора остаётся простая — иначе везде потянется тяжёлый запрос.

#### 2.3 Динамический список и запрос

- У динамического списка — **произвольный запрос**. Сначала запрос может тянуть только **справочник** или строить соединение с **остатками**.
- **Ошибка:** если **основная таблица** запроса — **остатки**, в списке окажется **только** номенклатура с ненулевым остатком; пользователь «потеряет» позиции, которые есть в справочнике, но без остатка — **неожидаемое** поведение.
- **Правильно:** основная таблица — **Справочник.Номенклатура**, к **остаткам** — **левое соединение**; в конструкторе это задаётся **галочками** (все строки справочника + подмножество с остатками).

#### 2.4 Язык запросов

- Язык запросов 1С **близок к SQL**; умение **читать** запросы критично для разработки.

#### 2.5 Назначение формы в документе

- В табличной части **«Расход товара»** у поля **номенклатура** задают **форму выбора** — вариант **«с количеством»**.

#### 2.6 ПриСозданииНаСервере и параметры формы

- У формы выбора — событие **«При создании на сервере»**; в **параметрах** формы (структура) должны появиться данные для отбора — в т.ч. **склад** из шапки документа.

#### 2.7 Основная таблица динамического списка

- Если основная таблица списка ошибочно оказалась **остатки**, в интерфейсе могут отображаться **«чужие»** иконки (как у регистра) — сигнал проверить **основную таблицу** и запрос.

#### 2.8 Динамическое считывание данных

- Галочка **«Динамическое считывание данных»**: при большом справочнике и большом числе остатков платформа **порционирует** выборку, подгружая окрестность **видимой** области — **экономия памяти**.

#### 2.9 Связи параметров выбора

- В документе настраивают **связи параметров выбора** с реквизитом **склад** шапки, чтобы в форму выбора передавался **контекст склада**.

#### 2.10 Остаток «по всем складам» vs по одному

- Если в запросе остаток получался как **сумма по всем складам**, нужно **параметризовать** виртуальную таблицу по **складу**.

#### 2.11 Виртуальная таблица «Остатки» регистра накопления

- Обращение к `РегистрНакопления.… .Остатки(…)` — это **виртуальная таблица**: под капотом сложный запрос к физическим таблицам и **срезам**.
- **Период** в параметрах: если **не задан** / пустой — берётся актуальный остаток «на сейчас» — одно из **быстрых** обращений; запрос остатка **на дату** может требовать **дополнительных** пересчётов.
- Условия по **измерениям** (например, склад) задают в **параметрах** виртуальной таблицы.

#### 2.12 Параметр запроса и список на форме

- В реквизитах формы у **списка** задают **параметр** (например, склад) и **передают** в него значение из **параметров формы** выбора.

#### 2.13 Поведение при очистке склада и режим «не очищать»

- При смене склака не всегда нужно **очищать** номенклатуру в строках — настраивают **режим** (в т.ч. **не изменять** связанные поля при изменении склада).

#### 2.14 Форма без переданного склада

- Если **параметр склада** не передан (форма открыта из другого места), запрос не должен «падать»: проверка через **`НайденноеЗначение`** / опциональность параметра, **`ВЫБОР`** в запросе: при **`Неопределено`** показывать **суммарный** остаток по **всем** складам; иначе — фильтр по складу.
- Конструктор запроса можно открыть и нажать **ОК** для **автоформатирования** текста.

### Блок 3: Параметры сеанса (25–35 минут)

#### 3.1 Зачем

- Кэшировать часто используемые данные **в рамках сеанса**; пример — **текущий пользователь** для полей «кто создал / изменил», ответственный.

#### 3.2 Пользователи ИБ и справочник

- **Пользователи информационной базы** — не **ссылочный** тип; для реквизитов документов нужен **Справочник.Пользователи** (или аналог): **код** при необходимости убрать из интерфейса и из **ввода по строке**, чтобы не получить **исключение**.
- Предопределённый элемент **«не авторизован»** — для сеанса без корректного пользователя.

#### 3.3 Параметр сеанса

- Создают параметр сеанса, например **«Текущий пользователь»**, тип — **СправочникСсылка.Пользователи**.

#### 3.4 Модуль сеанса

- В **модуле сеанса** — обработчик **«Установка параметров сеанса»**.
- Берут **имя** текущего пользователя ИБ (`ПользователиИнформационнойБазы.ТекущийПользователь` / по смыслу занятия), ищут элемент справочника **по наименованию**; если нет — **создают** элемент, вызывают **`Заполнить()`**, **`ПроверитьЗаполнение()`**, **`Записать()`**; при ошибках — **`ВызватьИсключение`** с понятным текстом.
- Предопределённый элемент используют для **стабильной** ссылки без поиска по строке.
- В конце: **`ПараметрыСеанса.ТекущийПользователь = …`**.

#### 3.5 Права и сеанс

- Код модуля сеанса выполняется в **служебном** контексте; при обращении к данным с проверкой прав ситуация может отличаться от «обычного» кода.
- Если у роли (например, **продавца**) **нет** права на справочник **«Пользователи»**, при чтении **параметра сеанса** в форме документа возможно **нарушение прав доступа** — ошибка видна в **журнале регистрации** (сортировка по ошибкам).
- Нужно выдать роли минимум **просмотр/чтение** по справочнику пользователей, если реквизиты ссылаются на него.

#### 3.6 Использование в документе

- В форме документа (например, **«Приход товара»**) поле **«Ответственный»** заполняют из **`ПараметрыСеанса.ТекущийПользователь`** при создании.

## Резюме лекции

### Ключевые моменты

1. **.cf** — только метаданные; полная выгрузка ИБ из конфигуратора — иные правила **монопольности** и консистентности; **СУБД** — отдельный сценарий бэкапа.
2. Методологии: реактивная поддержка без плана; **водопад** для повторяемых проектов; **Agile / Scrum** для изменчивых требований; **Kanban** для поддержки и потока задач.
3. **Динамический список**: основная таблица — **справочник**; к остаткам — **ЛЕВОЕ СОЕДИНЕНИЕ**; отдельная форма выбора «с остатком» только там, где нужно.
4. **Виртуальная таблица «Остатки»**: параметры периода и измерений; пустой период — типично «текущие» остатки; условия по складу — **параметризация**.
5. **Параметры выбора** и параметры **списка** связывают документ и запрос; при отсутствии склада — **ВЫБОР** / защита от **Неопределено**.
6. **Параметры сеанса** — ссылка на «своего» пользователя в справочнике; инициализация в **модуле сеанса**; нужны **права** на справочник у прикладных ролей.

---

### Практические рекомендации

1. Проверять запрос формы выбора: не исчезают ли позиции справочника из‑за **не того** соединения с остатками.
2. Включать **динамическое считывание** для больших списков.
3. При вводе параметров сеанса в формы учитывать **матрицу прав** и журнал ошибок.
