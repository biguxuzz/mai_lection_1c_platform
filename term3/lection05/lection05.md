# Лекция 5: Подсистема «Пользователи» и управление доступом (БСП)

**Продолжительность:** ~60–90 минут (фрагмент записи **27.03.2026** обрывается на демонстрации группы доступа; блок про поставляемые профили и код — по **ИТС** и материалам курса)  
**Целевая аудитория:** Студенты 3 курса, прошедшие курс разработки на платформе 1С:Предприятие (1–2 семестры)  
**Курс:** Библиотека стандартных подсистем  
**Источник занятия:** транскрипция `БСП_07_Пользователи_УправлениеДоступом_2026-03-27.txt` (демонстрация на демоконфигурации БСП; ориентир по редакции — **БСП 3.1.x / 3.2.x** и **ИТС**). Ранее: `БСП_08_Пользователи.txt`, `БСП_09_Управление_доступом.txt`.

## Цель лекции

Разобрать **обязательную для внедрения** подсистему **«Пользователи»**: кто такие **пользователи** и **внешние пользователи** в терминах БСП; **где** создавать пользователей при внедрённой БСП; какие **модули** дают методы **`ТекущийПользователь`**, **`ТекущийВнешнийПользователь`**, **`АвторизованныйПользователь`** и когда какой вызывать; что такое **служебные** пользователи и как с ними работать (пример из **серверных оповещений**, в т.ч. **фиксированный идентификатор** для однозначности между базами). Затем — подсистему **«Управление доступом»**: стандарт **«Настройка ролей и прав доступа»**, **атомарные роли**, особые требования к роли **«Полные права»**, **профили групп доступа**; как на практике собрать профиль для документа **«Реализация товаров и услуг»** и связанных справочников; когда **не** выдают права на **регистры накопления** и почему **нельзя** смешивать в одной атомарной роли объекты из **разных подсистем**; зачем нужны **базовые права БСП**, **запуск тонкого клиента** и **чтение НСИ** в демо; чем **профиль** отличается от **группы доступа**; типичная ошибка — **забыли роль на подсистему**; как **автоматизировать** поставляемый профиль через обработку **«Управление доступом»** и переопределяемый модуль *(по материалам ИТС, если на занятии не успели)*.

## Необходимые предзнания

- Состав БСП, публичный и переопределяемый API (лекции 1–2)
- Пользователи информационной базы, роли платформы (базовый курс по 1С)
- Справочники, документы, подсистемы конфигурации
- Навигация по инструкции БСП на **ИТС** (раздел по разработке, версия библиотеки)

## План лекции (~60–90 минут)

> **Примечание:** на занятии используется демоконфигурация БСП с документом **«Реализация товаров и услуг»**, справочниками **складов** и **складских помещений** и подсистемой **«Курс разработки»** (имя в вашей базе может отличаться).

### Блок 1: Подсистема «Пользователи» — зачем и что даёт (8–10 минут)

#### 1.1 Обязательность для внедрения

**Подсистема пользователей** — одна из последних обязательных для подключения при внедрении БСП: она обеспечивает работу со списками **пользователей** и **внешних пользователей**, синхронизируемых с **пользователями информационной базы**.

#### 1.2 Два типа субъектов

- **Пользователи** — пользователи **вашей** прикладной системы (в типовой модели — элементы справочника пользователей БСП).
- **Внешние пользователи** — в т.ч. **контактные лица организации**, сценарии **самообслуживания** и аналогичные модели.

#### 1.3 Создание пользователей при внедрённой БСП

Если в конфигурации подключена БСП, **пользователей для работы с прикладным учётом** нужно создавать **в режиме 1С:Предприятия** (через механизмы подсистемы), а **не** только через **администрирование** информационной базы: иначе пользователь **не попадёт** в справочник БСП, и **вход в приложение будет невозможен** (нет связи «пользователь ИБ ↔ элемент справочника»).

### Блок 2: Модули и методы «текущего» пользователя (10–12 минут)

#### 2.1 Модуль пользователя и модуль внешнего пользователя

**Модуль пользователя** предоставляет методы в духе **`ТекущийПользователь`** и **`АвторизованныйПользователь`**.  
**Модуль внешнего пользователя** — метод в духе **`ТекущийВнешнийПользователь`**.

#### 2.2 Поведение «текущий пользователь» vs «текущий внешний»

По смыслу занятия методы **похожи**, но **взаимно исключают** тип авторизации:

- **`ТекущийПользователь`** возвращает **ссылку** на элемент справочника пользователя; если в сеансе авторизован **внешний** пользователь — метод **вызывает исключение**.
- **`ТекущийВнешнийПользователь`** возвращает ссылку на элемент справочника **внешнего** пользователя; если авторизован **обычный** пользователь — **исключение**.

#### 2.3 Универсальный сценарий: «Авторизованный пользователь»

Если функционал должен работать **и** с пользователями, **и** с внешними — удобнее **`АвторизованныйПользователь`**: **не** бросает исключение, а возвращает ссылку на **соответствующий** справочник в зависимости от того, **кто** вошёл в систему.

---

### Блок 3: Служебные пользователи и пример «служебный пользователь» (10–12 минут)

#### 3.1 Назначение

**Служебные пользователи** — для **регламентных заданий**, **HTTP-сервисов** и иной **технической** работы от имени ИБ.

#### 3.2 Правила работы

В справочнике пользователя для таких субъектов есть признак (в речи — реквизит вроде **«Служебный»**): **создавать** таких пользователей нужно **программно** и **не менять** значение этого реквизита вручную в дальнейшем. **Свойства** такого пользователя нужно **отслеживать** в коде самостоятельно. Служебные пользователи **скрыты** из пользовательского интерфейса системы.

#### 3.3 Пример из кода: «СлужебныйПользователь»

В транскрипте отсылают к модулю **серверных оповещений**: метод **`СлужебныйПользователь`**.

**Логика (по смыслу демонстрации):**

- выполняется работа с **пользователями информационной базы** — **поиск**, при отсутствии — **создание**, проверка и при необходимости **обновление** свойств;
- затем **поиск** в справочнике пользователей с учётом признака **служебный**;
- при необходимости — **создание** элемента справочника и **запись**.

То есть метод **целиком обслуживает** связку: **пользователь ИБ** + **элемент справочника** пользователей БСП.

В актуальной поставке БСП создание пользователя **перенаправлено** в общий модуль **«Пользователи»**: в **описание пользователя** передаётся признак **служебный**, формируется структура, выполняется поиск в справочнике и при отсутствии — создание. Для **серверных оповещений** служебный пользователь **зафиксирован по GUID**: в разных информационных базах с БСП это **один и тот же** субъект (одинаковая **ссылка**), что удобно для переносимости и однозначной идентификации.

### Блок 4: Управление доступом — вводная и стандарт разработки (10–12 минут)

#### 4.1 Роль подсистемы

**Управление доступом** — одна из **самых сложных** подсистем БСП (вместе с тем задаёт **методологию** прав для прикладных решений): позволяет выдавать доступ **к таблицам** (объектам метаданных) и **к отдельным записям** в них (тема **ограничения на уровне записей** на занятии затрагивается **вскользь**, подробно — в следующей лекции). У роли **«Полные права»** / **«Администратор системы»** **нет** ограничений на уровне записей (**РЛС**); это **не** распространяют на «обычные» прикладные роли.

#### 4.2 Стандарт «Настройка ролей и прав доступа»

Типовые требования к проектированию ролей изложены в стандарте разработки на **1С:ИТС**: [«Настройка ролей и прав доступа»](https://its.1c.ru/db/v8std#content:689).

**Роли** проектируют **атомарными**: каждая даёт права на **элементарные** функции программы. Из атомарных ролей средствами БСП собирают **профили** доступа.

#### 4.3 Роль «Полные права» и новые объекты

По смыслу стандарта и демонстрации: право **«Устанавливать права для новых объектов»** и сопутствующие «опасные» флаги для **новых** объектов должны быть **только** у роли **«Полные права»**; **во всех остальных** ролях конфигурации эти права **должны отсутствовать**.

#### 4.4 Как проще спроектировать атомарные роли

В стандарте описано формирование ролей по **типам** объектов. Упрощённый приём из речи: **на каждый объект** завести **свою** роль (или набор ролей по шаблону стандарта), чтобы проще находить соответствие «объект ↔ роль» (например: **чтение реализации товаров и услуг**, **добавление изменения реализации товаров и услуг** — имена обсуждают с **руководителем разработки**: удобство поиска в конфигураторе vs грамматика формулировок вроде «чтение складов»).

**Для ссылочных объектов** (по стандарту и по приёму занятия): как минимум **две** роли — **чтение** и **добавление и изменение** (`…Чтение`, `…ДобавлениеИзменение` — по соглашению команды).

**Регистры накопления:** в **общем случае** отдельные роли на регистры **не выдают** — при **проведении** документа платформа **повышает** права; пользователю редко нужен прямой доступ к регистру. **Исключение:** объекты **настолько связаны**, что не имеют смысла по отдельности (пример из речи: документ заказов и связанный регистр) — тогда **оба** можно включить в **одну** атомарную роль.

**Запрет:** в **одну** атомарную роль **нельзя** объединять объекты из **разных подсистем** — при отключении одной подсистемы возникает **конфликт** и непредсказуемое поведение прав.

---

### Блок 5: Роли на подсистемы, отчёты и обработки (6–8 минут)

Помимо объектов данных, в стандарте учитывают:

- **подсистемы** — роль вида **«Подсистема» + имя подсистемы** (доступ к разделу интерфейса);
- **отчёты** — **просмотр отчёта** + имя;
- **обработки** — **использование обработки** + имя.

Это нужно, чтобы пользователь **видел** не только данные, но и **навигацию** и **инструменты**.

### Блок 6: Пример: профиль для документа «Реализация товаров и услуг» (15–18 минут)

#### 6.1 Две роли на документ

Для документа **«Реализация товаров и услуг»** добавляют, например:

- роль **чтения** (в речи — «Чтение, реализация товаров и услуг»; для поиска в конфигурации допустим именительный падеж имени);
- роль **добавления и изменения**.

**Настройка флагов** в роли (по демо): **снять лишние** флаги, затем для **чтения** — выставить нужную строку по документу; для **добавления/изменения** — выставить права **до «ввод по строке»**, затем **снять** **интерактивное удаление** и **«Удаление»**, если удаление не требуется (остаются изменение и ввод по строке).

#### 6.2 Справочники складов и складских помещений

Добавляют роли **чтения** и **добавление/изменение** для справочников **складов** и **складских помещений** с теми же принципами (в т.ч. **без удаления**, если так задумано).

#### 6.3 Подсистема «Курс разработки»

Выдают право на **подсистему** (в демо занятия **27.03.2026** — **«Курс разработки»**): **просмотр** подсистемы, чтобы раздел **отображался** в интерфейсе.

Роли, заведённые для объектов учебной подсистемы, добавляют в **состав** этой подсистемы в конфигураторе (чтобы права и навигация были согласованы).

#### 6.4 Обновление информационной базы после изменения ролей

После создания и правки **ролей** в конфигураторе конфигурацию запускают с **обновлением информационной базы** (типовой сценарий согласования метаданных с данными БСП), чтобы **новые права** стали доступны для настройки в режиме **1С:Предприятия** (профили, группы доступа).

### Блок 7: Профиль группы доступа в режиме предприятия (12–15 минут)

#### 7.1 Создание профиля

Путь (по демо): **Администрирование** → **Настройки пользователей и прав** → **Профили групп доступа** → **Создать**.

**Обязательно** включают типовые права входа:

- **«Базовые права БСП»**;
- **«Запуск тонкого клиента»** — чтобы пользователь мог **войти** в систему.

#### 7.2 Особенность демоконфигурации

Для работы **демонстрационной** конфигурации в профиль добавляют роль **«Демо: базовые права БСП»** (в речи — «демо базовые права БСП»): без неё приложение может **не запуститься** из‑за проверок параметров сеанса и базовых сценариев демо. Название роли уточняйте в своей поставке.

#### 7.3 Нормативно-справочная информация (НСИ)

Если документ использует справочники демо (например, **номенклатуру**), выдают роль **чтения НСИ** (в речи — **чтение нормативно-справочной информации**), если пользователь **не** редактирует справочники.

#### 7.4 Итоговый набор ролей в профиле «Продавец» (по смыслу демо)

В сумме: **базовые права БСП**, **запуск тонкого клиента**, **чтение НСИ**, **добавление/изменение** документа **«Реализация товаров и услуг»**, для **складов** и **складских помещений** — по сценарию занятия только **чтение**, роль **подсистемы** **«Курс разработки»**, а также **демо**-роль (**«Демо: базовые права БСП»** или аналог в вашей конфигурации).

**Профиль групп доступа** выступает **шаблоном**: его копируют/уточняют в **группах доступа**; **пользователям** назначают уже **группы доступа**, а не профили напрямую.

#### 7.5 Профиль vs группа доступа

Через профиль можно **преднастроить** ограничения, в т.ч. **на уровне записей**; в **группе доступа** их **доуточняют**. Группа доступа — **уточнение** профиля.

### Блок 8: Группа доступа, участники и проверка (8–10 минут)

#### 8.1 Создание группы

По гиперссылке **«Группы доступа»** создают группу (в демо профиль **«Продавец»** подставляется автоматически из параметров выбора). Указывают наименование (например, **«Наши продавцы»**).

#### 8.2 Участники

В табличной части **«Участники группы»** подбирают пользователей ИБ; при необходимости создают нового пользователя (в демо — отдельная учётная запись вроде **«Продавец 1»**). После записи группы пользователь получает **все роли** профиля.

#### 8.3 Проверка под ограниченным пользователем

Запуск **1С:Предприятия** под этим пользователем. Если **забыли** включить в профиль **роль подсистемы**, возможна ошибка в духе **«Авторизация не выполнена»** / **«Недостаточно прав для входа»** — без права на подсистему пользователь **не получает** доступ к разделам интерфейса, даже при наличии прав на объекты.

После добавления роли на **«Курс разработки»** в интерфейсе виден **соответствующий** раздел; доступны справочники и документ из этой подсистемы. Пользователь может **редактировать** документ, **проводить** — при выданных правах. **Изменять** справочники нельзя, если выдано только **чтение** (например, склады — только просмотр).

### Блок 9: Поставляемые профили и обработка «Управление доступом» (10–12 минут)

> **По записи 27.03.2026** демонстрация дошла до создания **группы доступа** и добавления пользователя; дальнейший разбор **обработки «Управление доступом»** и вставки кода в переопределяемый модуль — по **ИТС** и методичке курса (ниже — кратко, как в прежней версии лекции).

#### 9.1 Зачем автоматизировать

Чтобы **не настраивать** вручную одни и те же профили на каждой **тестовой** и **боевой** базе, настройку **поставляют** в конфигурации.

#### 9.2 Обработка из поставки БСП

Используют обработку **«Управление доступом»** (входит в **инструменты разработчика** поставки БСП): вкладка **«Начальное заполнение профилей»** → выбор профиля (**«Продавец»**) → **«Сформировать»** — получают **программный код** для вставки.

#### 9.3 Куда вставлять код

Код помещают в метод **`ПриЗаполненииПоставляемыхПрофилейГруппДоступа`** общего модуля **управления доступом переопределяемого** (точное имя модуля — по вашей поставке).

**Рекомендация из занятия:** не размещать всю логику в переопределяемом модуле — **вызвать** метод из **своей** библиотеки (в демо копируют модуль, например **`УправлениеДоступом_mp`** и переносят туда сгенерированный код).

#### 9.4 Доработка профиля после генерации

В сгенерированном коде могут быть **лишние** для прикладного решения роли (в демо убирают роль вроде **«Демо: базовые права из расширения»**). В параметрах обновления снимают флаг **«Запретить изменение профилей»** (**`Ложь`**), чтобы **разрешить** изменение поставляемых профилей — **на свой страх и риск** (как в речи преподавателя).

## Резюме лекции

### Ключевые моменты

1. Пользователей прикладной БСП создают **в режиме предприятия** (подсистема **«Пользователи»**), иначе нет записи в справочнике — **вход невозможен**. Подсистема синхронизирует **пользователей** и **внешних пользователей** с ИБ; для смешанных сценариев удобен **`АвторизованныйПользователь`**, чтобы не ловить исключения **`ТекущийПользователь`** / **`ТекущийВнешнийПользователь`**.
2. **Служебные** пользователи создаются **программно**; пример — **`СлужебныйПользователь`** (серверные оповещения): синхронизация ИБ и справочника; у служебного пользователя оповещений может быть **фиксированный GUID** для **одинаковой** ссылки в разных базах с БСП.
3. **Управление доступом** — сложная подсистема; роли **атомарные**, права **«для новых объектов»** — только у **«Полных прав»**; у **«Полных прав»** / администратора **нет** ограничений **РЛС** (это не про обычные прикладные роли). На **регистры накопления** в общем случае роли **не выдают**; в одну атомарную роль **не** смешивают объекты из **разных подсистем**.
4. После правок **ролей** в конфигураторе выполняют **обновление информационной базы**, затем настраивают **профиль** и **группу доступа** в режиме предприятия.
5. Профиль **«Продавец»** в демо: **базовые права БСП**, **тонкий клиент**, **«Демо: базовые права БСП»**, **чтение НСИ**, **добавление/изменение** **РТУ**, **чтение** складов/помещений, роль подсистемы **«Курс разработки»**. Без роли **подсистемы** возможна ошибка **авторизации** при входе.
6. Пользователям назначают **группы доступа**; профиль — **шаблон**, группа — **уточнение** (ограничения на уровне записей — следующая лекция).
7. Повторяемое заполнение профилей: обработка **«Управление доступом»** → код в **`ПриЗаполненииПоставляемыхПрофилейГруппДоступа`** → вызов из **своего** общего модуля; снятие **«Запретить изменение профилей»** — по необходимости и на свой риск.

### Практические шаги после лекции

1. На **ИТС** открыть стандарт [«Настройка ролей и прав доступа»](https://its.1c.ru/db/v8std#content:689) (v8std); затем раздел БСП: **пользователи**, **управление доступом** — по версии вашей библиотеки.
2. В демобазе проследить цепочку: роли в конфигураторе → **обновление ИБ** → **профиль** → **группа доступа** → вход под тестовым пользователем; убедиться, что в профиле есть роль на **нужную подсистему**.
3. Сформировать код в обработке **«Управление доступом»** и сравнить с переопределяемым модулем в конфигурации *(если проходили блок 9 по методичке)*.
