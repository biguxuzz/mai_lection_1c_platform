# Лекция 4: Подсистема «Обновление информационной базы» (БСП)

**Продолжительность:** ~45–50 минут  
**Целевая аудитория:** Студенты 3 курса, прошедшие курс разработки на платформе 1С:Предприятие (1–2 семестры)  
**Курс:** Библиотека стандартных подсистем  
**Источник занятия:** `БСП_05_Обновление_ИБ_2026-03-20.txt` (практическая демонстрация на демоконфигурации БСП; ориентир по редакции — **БСП 3.1.x / 3.2.x** и **ИТС**)

## Цель лекции

Разобрать **подсистему обновления версии информационной базы** в БСП: зачем нужны **обработчики обновления** при смене **реквизитного состава** уже существующих данных; как при **переносе логики** (пример: склад из шапки в табличную часть, **переименование** объекта) обеспечить **консистентность** и возможность работы со **старыми** документами; почему **применение обновления** меняет только **метаданные** и не выполняет прикладную миграцию; как при **пропуске релизов** выполняется **цепочка** обработчиков между версией в данных и целевой версией в конфигурации; как устроены **этапы** (в т.ч. **монопольный** этап до входа пользователей и **фоновая** доработка **больших объёмов**, в т.ч. **многопоточно**); как зарегистрировать **собственную подсистему** и **обработчики**, **отладить** их и безопасно **записывать** объекты через API БСП.

## Необходимые предзнания

- Состав БСП, публичный и переопределяемый API (лекции 1–2)
- Понятие **версии конфигурации** (редакция, подредакция, сборка)
- Работа с **общими модулями**, **модулями менеджера** документа, **запросами**
- Навигация по инструкции БСП на **ИТС** (раздел по разработке, версия библиотеки)

### Блок 1: Зачем нужна подсистема обновления ИБ (8–10 минут)

#### 1.1 Типовая бизнес-ситуация

**Пример из демонстрации (документ «Реализация товаров и услуг»):**

- В версии **2.x** логика опиралась на **склад в шапке** и на его основе строились движения.
- В версии **3.x** нужно **игнорировать** шапку и брать склад **из табличной части** (по строкам).
- Пользователь может **перепровести старый** документ, где склад ещё был в шапке — данные должны оставаться **согласованными**; для этого нужны **обработчики переноса** реквизитов.

**Другой пример из речи:** **переименование** справочника (ошибка в имени «доехала» до прода) — метаданные исправили, но **данные** и ссылки требуют осмысленной обработки в обработчиках.

**Проблема после простого «применения обновления» конфигурации:**

- Обновление меняет в первую очередь **метаданные**; **реквизитный состав** данных в базе **сам по себе** типовыми средствами не «переезжает» в новую структуру.
- Новые поля табличной части могут остаться **пустыми**, старые значения в шапке (в т.ч. у **удаляемого** реквизита вроде **«УдалитьСклад»**) — **заполнены**.
- Нужен **прикладной** код в **обработчиках обновления** в момент смены версии (и при необходимости — повторяемый, **идемпотентный** сценарий: при обрыве обработки дорабатываются только необработанные объекты).

#### 1.2 Роль подсистемы БСП

**Подсистема обновления версии ИБ:**

- Фиксирует в **данных**, какая **версия подсистемы** уже применена к информационной базе, и сопоставляет её с версией в **конфигурации**.
- При переходе на новую версию **запускает цепочку обработчиков** между «было» и «надо».
- Если пользователь **давно не обновлялся** (например, был на **2.0**, а релизы выходили **ежемесячно**), при переходе на **3.0** должны выполниться **все** промежуточные обработчики на пути от фактической версии данных до целевой — иначе данные будут **неконсистентны**.

**Этапы процесса (вводно по записи занятия):**

1. **Монопольный** этап — то, без чего при работе с **необновлёнными** данными будут **ошибки**; выполняется **до** полноценной работы пользователей с ИБ (после обновления конфигурации).
2. Часть обработок можно выполнить **уже после** начала работы пользователей — **в фоне**, в т.ч. **в несколько потоков**.
3. Большой объём **исторических** реализаций (перенос склада из шапки в строки) не обязан блокировать базу на часы: его выносят в **фон**, пока пользователь вводит **новые** оперативные данные.

**Итог:** обновление — это не только загрузка новой структуры метаданных, но и **прикладная миграция данных** под новую логику.

### Блок 2: Версии подсистем и «основная конфигурация» (6–8 минут)

#### 2.1 Регистр сведений «Версии подсистем»

Открывают регистр сведений **«Версии подсистем»** (в интерфейсе — по смыслу «что зафиксировано в **данных** по версиям подсистем»).

**В регистре видно:**

- какие **подсистемы** используются в информационной базе и какие у них **версии в данных**;
- флаг **основной конфигурации**: выставляется, когда **имя подсистемы совпадает с именем конфигурации** (в демо: конфигурация вроде **«Библиотека стандартных подсистем …»** и одноимённая подсистема помечена как **основная**).

**Зачем версия дублируется в данных, если в конфигурации уже есть версия подсистемы:** чтобы при запуске сравнить «что уже **прогнали** в этой ИБ» и «до какой версии **хотим** прийти» и **вычислить список обработчиков** для всех промежуточных шагов. Без актуальной записи в регистре система не знает, с какой точки продолжать миграцию.

**Редактирование версии в регистре** (включить возможность редактирования) используют для **экспериментов** и **повторного** запуска пропущенных обработчиков: версию в данных **понижают** — при следующем обновлении цепочка может выполниться снова.

#### 2.2 Почему важен запуск «обновление информационной базы»

Для отработки обработчиков и актуализации регистра версий нужно уметь **инициировать процесс обновления ИБ** (в транскрипте — «запустить обновление информационной базы»; точная формулировка **параметра запуска** — по документации к вашей версии платформы и БСП).

Без прохождения этого процесса **запись в регистре** может **не соответствовать** целевой версии, и **обработчики не выполнятся**, даже если версию в конфигурации уже изменили.

### Блок 3: Своя «библиотека» и модуль обновления (12–15 минут)

#### 3.1 Шаблон общего модуля

**Типовой путь (как на занятии):**

1. Взять за основу модуль **`ОбновлениеИнформационнойБазыБСП`** (в речи — «обновление базы БСП»; точное имя зависит от поставки).
2. **Скопировать** модуль.
3. Именовать новый модуль по **правилам именования**: часто используют **буквы от названия своей подсистемы** (в демо — подсистема вроде **«Курс разработки»** → сокращение в имени модуля обновления).
4. **Очистить** шаблон от лишнего: убрать копирайт и ненужные на старте реализации; **последние три** процедуры/блока в типовом шаблоне могут понадобиться **позже** — ориентируйтесь на комментарии к вашей поставке БСП.

#### 3.2 Процедура «При добавлении подсистемы»

В структуре описания подсистемы задать:

- **`Имя`** — имя вашей подсистемы (в демо — **«Курс разработки»**).
- **`Версия`** — строка из **четырёх разрядов** `A.B.C.D` (семантика по записи занятия):
  - **1-й разряд** — **крупные** изменения конфигурации, «глобально» новое в релизе;
  - **2-й разряд** — **существенное** изменение бизнес-логики, поведения объектов;
  - **3-й разряд** — изменилась логика, но **без** сильной перестройки объектов;
  - **4-й разряд** — по сути **номер сборки** / мелкие приращения.

**Зависимости:**

- **`Требуемые подсистемы`** — массив; для работы обязательно указать **библиотеку стандартных подсистем** (БСП).

#### 3.3 Снятие с поддержки и подключение переопределяемого модуля

После копирования кода модуль конфигурации может быть **на поддержке** — для правок переопределяемых модулей нужно **изменить правила поддержки** (снять запрет редактирования) для соответствующих объектов — иначе правки не сохранить.

**Модуль:** `ПодсистемыКонфигурацииПереопределяемый` (в речи — «подсистемы конфигурации переопределяемые»; уточняйте имя в вашей поставке).

**Процедура:** **`ПриДобавленииПодсистем`**:

- в массив модулей подсистем добавить **имя** **вашего** общего модуля обновления (по смыслу тот же слой, что типовой **`ОбновлениеИнформационнойБазыБСП`**, но с вашим именем после копирования).

Аналогичное добавление может требоваться в **серверных общих модулях** в контексте **«модули подсистемы»** — следуйте структуре демо-конфигурации и ИТС.

**Рекомендация БСП:** в переопределяемом модуле **не размещать** объёмную логику — только **вызов** одноимённого метода из вашего общего модуля.

#### 3.4 Регистрация модуля в составе подсистем

В **составе подсистем** конфигурации нужно зарегистрировать **имя** нового общего модуля обновления (как в демо при «добавлении подсистемы»).

#### 3.5 Проверка после запуска с параметром обновления

После запуска с параметром обновления ИБ в регистре **«Версии подсистем»** должна появиться строка **вашей** библиотеки с указанной **версией**.

### Блок 4: Обработчики обновления (10–12 минут)

#### 4.1 Процедура «При добавлении обработчиков обновления»

В общем модуле вашей библиотеки — метод **`ПриДобавленииОбработчиковОбновления`** (в документации БСП — с подробным **примером** в комментарии к типовому модулю).

**Обработчики** задаются через **таблицу значений** с колонками, описанными в БСП (см. также структуру **`НоваяТаблицаОбработчиковОбновления`** / аналог в вашей версии).

#### 4.2 Колонка «Версия»

- В колонке задаётся версия подсистемы, **до которой** выполняется переход с **данным** обработчиком (целевая точка на пути обновления).
- **Пример из речи:** в регистре в данных ещё **1.0.1.0**, в обработчиках объявлен переход на **1.0.2.1**; обработчик с версией **1.0.1.5** **меньше**, чем **1.0.2.1**, и должен быть выполнен на пути к целевой версии (конкретные правила отбора — по таблице обработчиков и документации БСП).

**Отладка:** в данных/комментариях к строке обработчика можно указать пометку вроде **«Отладка обработчика»**, чтобы проще находить сценарий при отладке.

#### 4.3 Указание процедуры-обработчика

В обработчике указывается **метод**, который будет вызван. Логично размещать код **не в модуле объекта**, а в **модуле менеджера** того объекта конфигурации, к которому относится миграция (в демо — документ **«Реализация товаров и услуг»**, процедура в духе **`ЗаполнитьТабличныеЧастиПриОбновлении`** / перенос склада в табличную часть).

Процедура должна быть **`Экспорт`** — иначе подсистема обновления её не «увидит» из другого модуля.

**Типичная ошибка из демо:** опечатка в **имени процедуры** в таблице обработчиков → ошибка вида **неверный формат параметра «имя процедуры»**; путаница **`Документы.…`** vs **`документы.…`** (регистр/написание) → **«не существует менеджера объекта»**.

#### 4.4 Режим выполнения (вводно)

В таблице обработчиков задаётся **режим выполнения**:

- **монопольно**;
- **оперативно**;
- **отложено**.

На первом проходе в демо выбран режим **монопольный**; детальный разбор — в конце лекции.

#### 4.5 Обработчики «на любую версию»

Для обработчиков, которые должны выполняться **при каждом** обновлении подсистемы **независимо** от номера версии, в БСП допускается указание версии **«*»** (звёздочка). *(Уточняйте поведение по инструкции вашей редакции БСП.)*

### Блок 5: Пример кода обработчика (перенос склада) (12–15 минут)

#### 5.1 Алгоритм в модуле менеджера документа

**Задача:** для уже существующих документов:

- в **табличной части** «Товары» склад **не заполнен** (пустая ссылка);
- при этом в **удаляемом** реквизите шапки **«УдалитьСклад»** значение **есть**.

**Запрос:**

- выбрать **уникальные ссылки** на документы (**«Различные»** / **DISTINCT**): при обращении к табличной части одна и та же ссылка на документ может повторяться по числу строк;
- условия: шапка **УдалитьСклад** заполнена, в строке табличной части склад **пустой** — чтобы при повторном запуске обрабатывать только **ещё не перенесённое** (**идемпотентность**).

**Подсказка типизации в коде:** платформа может не подсказывать свойства «на пустом» `ДокументОбъект`; приём из демо — временно создать переменную нужного типа документа в **недостижимой ветке** `Если Ложь`, чтобы среда знала тип и подсказала реквизиты.

#### 5.2 Обход и запись

Для каждой ссылки:

- получить **объект документа**;
- для строк табличной части, где склад пустой, проставить склад из **`ДокументОбъект.УдалитьСклад`** (или актуальное имя удаляемого реквизита); если в строке склад уже заполнен — **ничего не делать**;
- **записать** документ через API БСП (см. ниже), а не «голой» записью, если важно не задеть обмен.

#### 5.3 Отключение побочных эффектов при записи

Так как **логика проведения** по смыслу **не меняется** (данные лишь «раскладываются» по новой структуре), важно **не вызвать** лишние обработчики, обмен, интеграции:

- **`ОбменДанными.Загрузка = Истина`** — типовой приём подавления части прикладной логики при программной записи;
- **`ДокументОбъект.ОбменДанными.Получатели.АвтоЗаполнение = Ложь`** — чтобы объект **не зарегистрировался** автоматически на планах обмена с авторегистрацией.

**Упрощение:** в модуле обновления БСП есть готовый метод **`ЗаписатьОбъект`** (в транскрипте — «записать объект» из API подсистемы) — он **инкапсулирует** типовые настройки обмена для сценария обновления. В демо передают **документ-объект** в этот метод.

### Блок 6: Отладка и повторный запуск обработчика (5–7 минут)

#### 6.1 Тестовые данные

В пользовательском режиме создать ситуацию: в строках **нет** склада, в шапке (удаляемый реквизит) склад **есть**; при необходимости **не проводить** документ, если проверка заполнения мешает (в демо всплыла ошибка: сравнение с **Истина** для признака, который может быть **Неопределено** — нужно явно обработать **Неопределено** / пустую ссылку до логики с «складом с помещениями» и т.п.).

#### 6.2 Отладка фоновых сценариев

Обновление ИБ на **сервере** выполняется **фоновым заданием** — для пошаговой отладки включают **режим отладки** в **сервис → параметры** (аналогично тому, как настраивали на курсе по платформе). Не забыть **точку останова** в обработчике.

**На семинаре** планируется разбор **более сложного** сценария, чем учебный монопольный обработчик с лекции.

#### 6.3 Повторный запуск того же обработчика

**Вариант 1:** в регистре **«Версии подсистем»** **понизить** версию своей библиотеки (включить редактирование записи, ввести, например, **1.0.1** вместо более новой) — при следующем обновлении цепочка может выполниться **снова**.

**Вариант 2:** временно в **`ПриДобавленииОбработчиковОбновления`** подставить версию вроде **«ОтладкаОбработчика»** / ослабить условие — только для **dev**; в бою так не оставлять.

### Блок 7: Режимы выполнения: монопольный, оперативный, отложенный (6–8 минут)

*(Сверка с ИТС по вашей редакции БСП; ниже — связка с вводной частью записи `БСП_05`.)*

#### 7.1 Монопольный режим

- Выполняется **до появления пользователей** в базе там, где иначе система при работе с **необновлёнными** данными даст **ошибки**.
- В демо первый обработчик переноса склада сделан **монопольным** как учебный пример критичного шага.

#### 7.2 Оперативный режим

- Используется в сценариях, где обработка должна завершиться в **оперативном** контуре обновления (подробности и взаимодействие с монопольным этапом — **ИТС**).
- Если на пути обновления есть **монопольные** обработчики, **оперативная** часть может **упираться** в общие правила порядка выполнения (уточняйте по редакции БСП).

#### 7.3 Отложенный режим и фон

- Часть обработок выполняется **после** начала работы пользователей — **в фоне**, что снижает **простой** при **больших** объёмах **исторических** данных.
- Такие задания могут выполняться **в несколько потоков** (**мультипоточно**), чтобы **ускорить** миграцию.
- В типовых конфигурациях для «тяжёлых» объёмов **часто** выбирают именно **отложенную** обработку.

## Резюме лекции

### Ключевые моменты

1. При изменении структуры и логики (пример — перенос склада из шапки в ТЧ) **одного обновления КД недостаточно**: нужны **обработчики обновления**, которые перенесут данные в момент смены версии.
2. Подсистема БСП отслеживает **версии** и запускает обработчики по правилам **диапазонов версий**.
3. **«Версии подсистем»** — ключевой регистр **версий в данных**; по нему вычисляется, **какие** обработчики прогнать до целевой версии в конфигурации; признак **основной конфигурации** выставляется, когда **имя подсистемы совпадает с именем конфигурации**; для отработки цепочки нужен запуск процесса **обновления ИБ**.
4. Своя подсистема: копия **`ОбновлениеИнформационнойБазыБСП`** → своё имя модуля, заполнение **`ПриДобавленииПодсистем`**, **`ТребуемыеПодсистемы`** (БСП), правки в **`ПодсистемыКонфигурацииПереопределяемый`**, при необходимости **снятие с поддержки** для редактирования.
5. Обработчики регистрируются в **`ПриДобавленииОбработчиковОбновления`**; версия определяет **момент** запуска; есть режим **«*»** для повторяемых сценариев (по документации).
6. При записи из обработчика удобно вызывать **`ЗаписатьОбъект`** из API подсистемы обновления БСП — он выставляет **`ОбменДанными.Загрузка`**, отключает автозаполнение получателей обмена и т.д. (аналогично для регистров при миграции движений).
7. **Монопольный** этап — то, без чего нельзя пускать пользователей; **фон** и **отложенные** режимы — большие объёмы и **параллельное** выполнение без длительной блокировки базы.
