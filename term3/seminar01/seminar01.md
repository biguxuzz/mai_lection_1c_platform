# Семинар 1: Методики работы с БСП и базовая функциональность

**Продолжительность:** ~55 минут  
**Целевая аудитория:** Студенты 3 курса, прошедшие курс разработки на платформе 1С:Предприятие (1–2 семестры)  
**Курс:** Библиотека стандартных подсистем

## Цель семинара
Познакомить студентов с методиками работы, рекомендованными БСП (переменные параметры приложения, параметры сеанса, обновление вспомогательных данных, переименование объектов), с программным интерфейсом подсистемы «Базовая функциональность»: проверка заполненности с условными реквизитами, работа с динамическим списком и отбором, а также с выполнением длительных операций через фоновые задания.

## Необходимые предзнания
- Лекция 1: введение в БСП, обязательные подсистемы, приёмы интеграции
- Демонстрационная конфигурация БСП с собственными объектами (документы «Приобретение» и «Реализация товаров и услуг», справочники «Склады», «Складские помещения»)
- Работа с модулем объекта документа, формой списка, обработкой заполнения

### Блок 1: Методики работы, рекомендуемые БСП (15 минут)

#### 1.1 Переменные параметры приложения вместо глобальных переменных (3 минуты)

**Рекомендация БСП:** вместо множества глобальных переменных использовать **ПараметрыПриложения** — единое хранилище глобальных значений.

**Инициализация:**
- Формировать имя параметра, например: «СтандартныеПодсистемы» (или имя вашей библиотеки/подсистемы) + имя параметра
- Проверять, инициализирован ли параметр; если нет — выполнить вставку (инициализацию значения)
- В дальнейшем получать и устанавливать значения через эту переменную

**Контекст:** подсистема «Базовая функциональность» предоставляет универсальные механизмы запуска и завершения работы конфигурации, процедуры и функции общего назначения, роли (полные права, базовые права БСП и т.д.), а также методики доработки конфигурации.

---

#### 1.2 Параметры сеанса: отложенная инициализация (4 минуты)

**Рекомендация:** инициализировать параметры сеанса не при старте системы, а **при первом обращении** к параметру. Иначе замедляется старт, а параметр может быть не востребован.

**Реализация:**
- Использовать модуль **«ОбщегоНазначенияПереопределяемый»** и метод **«ПриДобавленииОбработчиковУстановкиПараметровСеанса»**
- Метод задаёт соответствие: имя параметра сеанса → обработчик установки
- Обработчик (например, в модуле менеджера справочника) имеет два параметра: **имя параметра** и **устанавливаемые параметры** (массив строк имён параметров сеанса, которые были установлены)

**Ранняя инициализация (если отложить нельзя):**
- В том же модуле «Общего назначения переопределяемый» — метод **«ПередЗапускомПрограммы»** для инициализации параметров сеанса до старта сеанса

- Используются для клиентского кода при запуске и завершении приложения (описания событий см. в комментариях модуля)

#### 1.4 Обновление вспомогательных данных (4 минуты)

**Назначение:** в информационной базе хранятся вспомогательные данные (дерево метаданных, разобранные правила обмена и т.д.). Их нужно обновлять после изменения метаданных.

**Способы запуска обновления:**
1. **Параметр запуска 1С:** Сервис → Параметры → Запуск 1С:Предприятие → параметр **«ЗапуститьОбновлениеИнформационнойБазы»**
2. **Стартер:** Изменить базу → Дополнительные параметры запуска → тот же параметр
3. **Изменение версии конфигурации** (в рамках подсистемы обновления ИБ — отдельная тема)
4. **Обработка «ОбновлениеВспомогательныхДанных»** (в поставке БСП, в инструментах разработчика): обновить все подсистемы, либо указать конкретную подсистему или таблицы

**Пример:** справочник **«ИдентификаторОбъектовМетаданных»** (базовая функциональность) хранит сведения о метаданных. После добавления новых справочников/документов в конфигурацию в этом справочнике их не будет, пока не выполнено обновление вспомогательных данных. После обновления — элементы появятся.

---

#### 1.5 Переименование объектов метаданных (2 минуты)

**Сценарии:**
- **Документ/справочник переименован:** имя в «Идентификаторе объектов метаданных» обновится при следующем обновлении (идентификатор-ссылка остаётся тем же).
- **Справочник помечен на удаление + создан новый с тем же именем:** старый идентификатор в справочнике «Идентификатор объектов метаданных» начнёт ссылаться на новый справочник (удобно при реструктуризации с переносом данных).
- **Подсистема переименована:** нужно явно указать переименование в модуле **«Общего назначения переопределяемый»** (настройки поддержки → переопределяемый модуль) — в комментариях приведён пример вставки: «До переименования» → «После переименования». Для срабатывания переименования подсистемы необходимо изменить версию конфигурации и выполнить обновление.

**Важно:** в общих модулях БСП без постфикса «Переопределяемый» или без явного разрешения на изменение — код не меняем. Редактируем только переопределяемые модули.

### Блок 3: Проверка заполненности и работа со списком (20 минут)

#### 3.1 Условная проверка заполненности (7 минут)

**Шаг 1 — свойство реквизита:** на реквизите «Складское помещение» (табличная часть «Товары») установить свойство **«Проверка заполнения»** = **«Выдавать ошибку»**.

**Шаг 2 — исключить реквизит из проверки при невыполнении условия:** в модуле объекта документа, в обработчике **ПроверкаЗаполнения**, в список непроверяемых реквизитов добавить наш реквизит:

```1c
Общего назначения.УдалитьНепроверяемыеРеквизитыИзМассива(ПроверяемыеРеквизиты, НепроверяемыеРеквизиты);
// В НепроверяемыеРеквизиты добавить: "Товары.СкладскоеПомещение"
```

Далее в коде проверки заполнения для каждой строки табличной части: если у склада установлен флаг «Использовать складские помещения» и не заполнено складское помещение — сформировать ошибку и установить флаг отказа.

**Ошибка:** обращение к складу через точку (СтрокаТовары.Склад.ИспользоватьСкладскиеПомещения) приводит к загрузке всего объекта справочника в память (запрос «Выбрать * from Справочник.Склады где Ссылка = &Ссылка»). Это неоптимально.

**Правильно:** получить один реквизит через **Общего назначения.ЗначениеРеквизитаОбъекта(СтрокаТовары.Склад, "Использовать складские помещения")**. В справочнике «Склады» должен быть реквизит «Использовать складские помещения» (булево).

---

#### 3.2 Устранение запросов в цикле (5 минут)

**Проблема:** в цикле по строкам табличной части для каждой строки вызывать ЗначениеРеквизитаОбъекта — значит выполнять запрос в цикле.

**Решение:** один раз получить значения реквизита для всех складов табличной части:
- Выгрузить колонку ссылок на склады: `СкладыСсылки = Объект.Товары.ВыгрузитьКолонку("Склад")`
- Вызвать **Общего назначения.ЗначенияРеквизитовОбъектов(СкладыСсылки, "Использовать складские помещения")**
- Возвращается **Соответствие**: ключ — ссылка из массива, значение — значение реквизита
- В цикле по строкам: `ИспользоватьСкладскиеПомещения = СоответствиеЗначений.Получить(СтрокаТовары.Склад)`

**Расширение:** метод **ЗначенияРеквизитовОбъекта** (множество реквизитов) — можно передать имена реквизитов строкой через запятую, массивом или структурой (ключ — псевдоним поля, значение — путь к данным, например «Владелец.ИспользоватьСкладскиеПомещения»).

---

#### 3.3 Сообщение пользователю об ошибке заполнения (3 минуты)

**Формирование текста и вывод:**
- Текст сообщения в стиле платформы: **Общего назначенияКлиентСервер.ТекстОшибкиЗаполнения(ВидПоля, ВидСообщения, ИмяПоля, НомерСтроки, ИмяСписка, ТекстСообщения)**
  - Вид поля: поле / колонка / список — для табличной части указываем «колонка»
  - Вид сообщения: заполнение / корректность — в нашем случае «заполнение»
  - Имя поля: «Складское помещение», номер строки — СтрокаТовары.НомерСтроки, имя списка — «Товары»
- Вывод: **Общего назначения.СообщитьПользователю(ТекстСообщения, КлючДанных, Поле, ВозвратОтказ)**
  - Ключ данных: ЭтотОбъект (в модуле объекта)
  - Поле: полный путь к ячейке — **Общего назначенияКлиентСервер.ПутьКТабличнойЧасти("Товары", СтрокаТовары.НомерСтроки, "СкладскоеПомещение")**
  - ВозвратОтказ — параметр по ссылке для установки отказа

После реализации: два склада (один с флагом «Использовать складские помещения», другой без). При проведении документа с пустым складским помещением в строке с первым складом — ошибка; во второй строке — без ошибки.

#### 3.4 Автор документа и отбор «Только мои документы» (5 минут)

**Реквизит документа:** «Автор» (тип: Справочник.Пользователи — из подсистемы «Пользователи»).

**Заполнение:** в обработчике **ОбработкаЗаполнения** (вызывается для новых документов, при вводе на основании и при программном Заполнить):  
`Автор = Пользователи.ТекущийПользователь();`

**Форма списка:**
- Реквизит формы (булево): «Только мои документы», вынести на форму
- Обработчик «При изменении» реквизита (клиент + процедура на сервере): установить/снять отбор по полю «Автор»

**Установка отбора:**  
**Общего назначенияКлиентСервер.УстановитьЭлементОтбораДинамическогоСписка(Список, "Автор", Пользователи.ТекущийПользователь(), ВидСравненияКомпоновкиДанных.Равен, , ТолькоМоиДокументы)**

- Список — реквизит формы (динамический список)
- Имя поля отбора — «Автор»
- Правое значение — текущий пользователь
- Вид сравнения — Равен
- Использование — значение реквизита «Только мои документы»

**Альтернатива:** не заводить свой реквизит, а при «При создании на сервере» формы списка добавить элемент отбора в пользовательские настройки компоновщика (режим отображения, например «Быстрый доступ»). Результат для пользователя аналогичен.

**Важно:** при наложении отборов нужно понимать, как они сочетаются; для отладки полезно получать текст запроса динамического списка (см. ниже).

---

#### 3.5 Отладка запроса динамического списка (по желанию)

**Идея:** реализовать свой метод (например, в общем модуле «Общего назначения мой проект»), который по переданному элементу формы типа «Таблица формы» (динамический список) возвращает выполняемый запрос.

**Шаги:**
- Проверить тип параметра: таблица формы для динамического списка (Общего назначенияКлиентСервер.ПроверитьПараметр)
- У элемента списка вызвать **ПолучитьИсполняемуюСхемуКомпоновкиДанных** → схема и настройки компоновщика
- Сформировать макет компоновки (КомпоновщикМакетаКомпоновкиДанных), получить наборы данных, из набора — текст запроса и значения параметров
- Вернуть объект Запрос с установленными параметрами

**Применение:** в режиме отладки вычислить этот запрос, при необходимости вывести его в консоль запросов. Преобразование в XML: **Общего назначения.ЗапросВСтрокуXML(Запрос)** — скопировать в консоль запросов с подставленными параметрами для отладки.

### Блок 4: Длительные операции (15 минут)

#### 4.1 Зачем выполнять длительные операции асинхронно (3 минуты)

**Правило:** длительные серверные вызовы (более ~8 секунд в типовых сценариях) не должны выполняться синхронно из клиентского кода. К таким операциям относятся: формирование отчётов, групповые обработки, загрузка/выгрузка данных, заполнение больших табличных частей.

**Риски синхронного выполнения:**
- Потеря работоспособности приложения, «зависание» интерфейса
- Браузер может предложить остановить сценарий
- Веб-сервер может вернуть 504 (шлюз не отвечает)
- Пользователь не может отменить операцию

**Подход БСП:** код длительной операции размещается в **модуле менеджера** объекта или в **общем модуле**; результат помещается во **временное хранилище**. Запуск — через фоновое задание; на клиенте используется механизм **длительной операции** (не фоновое задание напрямую, т.к. тонкий клиент не имеет доступа к фоновым заданиям, но длительная операция возвращает структуру со статусом, идентификатором задания, адресом результата для ожидания на клиенте).

---

#### 4.2 Пример: заполнение табличной части документа (7 минут)

**Команда на форме документа:** «Заполнить табличную часть». Свойство команды **«Изменяет сохраняемые данные»** = Истина (чтобы при «Только просмотр» команда была недоступна и при выполнении устанавливалась модифицированность формы).

**Неправильно:** реализовать длительный метод в модуле формы — его нельзя вызвать извне (например, программно из другого места).

**Правильно:**
1. Метод заполнения таблицы — в **модуле менеджера** документа, экспортный. Параметры: например, дата документа и длительность имитации (в секундах). Внутри: запрос или расчёт → таблица значений → при необходимости имитация задержки (цикл по времени). Результат — поместить во **временное хранилище** и вернуть адрес (или саму таблицу для записи во временное хранилище в БСП это часто делается внутри подсистемы длительных операций).
2. На сервере формы: процедура по команде не вызывает этот метод напрямую, а вызывает функцию, которая **запускает длительную операцию** и возвращает её описание.
3. **ДлительныеОперации.ВыполнитьФункцию(ПараметрыВыполненияФункции(ИдентификаторФормы), "Документы.РеализацияТоваровУслуг.ТаблицаЗаполненияТоваров", Объект.Дата, 8)** — возвращает структуру длительной операции (статус, идентификатор задания, адрес результата и т.д.).
4. На клиенте: **ДлительнаяОперацияКлиент.ОжидатьЗавершения(ДлительнаяОперация, ОповещениеОЗавершении(ЗавершениеЗаполненияТабличнойЧасти, ЭтотОбъект), ПараметрыОжидания)**.
5. **ПараметрыОжидания:** форма-владелец, заголовок и текст сообщения («Выполняется получение данных…»), при необходимости — оповещение пользователю (Показать = Истина, Текст, Пояснение) для уведомления после завершения.
6. В обработчике **ЗавершениеЗаполненияТабличнойЧасти(Результат, ДополнительныеПараметры):** проверить Результат на Неопределено (отмена пользователем); проверить статус длительной операции на ошибку; при успехе — по адресу результата из временного хранилища загрузить таблицу и вызвать серверный метод **ЗаполнитьТоварыИзВременногоХранилища(Адрес)** для загрузки табличной части.

**Итог:** пользователь нажимает «Заполнить табличную часть» → видит окно ожидания → по завершении — оповещение и обновлённая табличная часть на форме.

---

#### 4.3 Отладка длительных операций (2 минуты)

- **Подключение к отладке фоновых заданий** (соответствующий флаг в настройках) — можно отлаживать код, выполняемый в фоновом задании.
- **Режим отладки в параметрах запуска:** в параметре «Запуск» указать режим отладки; при включённой конфигурации с этим режимом БСП может выполнять длительные операции **в том же потоке**, что и приложение (без реального фонового задания), тогда срабатывают точки останова в модуле менеджера.

---

#### 4.4 Выполнение в несколько потоков (3 минуты)

**Методы:** ДлительныеОперации.**ВыполнитьФункциюВНесколькоПотоках**, ВыполнитьПроцедуруВНесколькоПотоках.

**Отличие от ВыполнитьФункцию:** вместо одного набора параметров передаётся **Соответствие**: ключ (например, номер потока) → массив параметров для вызова функции в этом потоке. В **адресе результата** возвращается не один результат, а **Соответствие**: ключ → результат выполнения функции для этого потока.

**Обработка результата:** в обработчике завершения обойти соответствие по ключам, для каждого ключа взять адрес результата из временного хранилища, при необходимости обработать ошибку (ОбработкаОшибок.КраткоеПредставлениеОшибки), затем дополнить табличную часть документа (например, Общего назначенияКлиентСервер.ДополнитьТаблицу(ТаблицаИсточник, ТаблицаПриемник)).

**Параметр «Ожидать завершения»:** если в параметрах выполнения указано Неопределено, клиентский код будет ждать завершения всех фоновых заданий, после чего можно обрабатывать результат в том же методе. Пример использования в БСП — рассылка отчётов (разбивка по порциям/потокам).

## Резюме семинара (5 минут)

### Ключевые моменты

1. **Методики БСП:**
   - Переменные параметры приложения вместо глобальных переменных
   - Отложенная инициализация параметров сеанса (модуль «Общего назначения переопределяемый»)
   - Обновление вспомогательных данных (параметр запуска, обработка, справочник «Идентификатор объектов метаданных»)
   - Переименование объектов и подсистем — с учётом правил БСП и переопределяемых модулей

2. **Базовая функциональность:**
   - Использование программного интерфейса (Общего назначения, ЗначениеРеквизитаОбъекта, ЗначенияРеквизитовОбъектов, УдалитьНепроверяемыеРеквизитыИзМассива, ТекстОшибкиЗаполнения, СообщитьПользователю, ПутьКТабличнойЧасти, УстановитьЭлементОтбораДинамическогоСписка)
   - Условная проверка заполненности без запросов в цикле
   - Отбор «Только мои документы» и отладка запроса динамического списка

3. **Длительные операции:**
   - Длительные серверные вызовы (> ~8 сек) выполнять асинхронно (фоновое задание через подсистему длительных операций)
   - Код — в модуле менеджера, результат — во временное хранилище
   - На клиенте: ОжидатьЗавершения, обработчик оповещения, загрузка из временного хранилища
   - Отладка: флаг отладки фоновых заданий или режим отладки в параметрах запуска
   - ВыполнитьФункциюВНесколькоПотоках — соответствие параметров и результатов по ключам

---

### Практические рекомендации

1. **Методики:** инициализировать параметры сеанса при первом обращении; после добавления/переименования объектов выполнять обновление вспомогательных данных; править только переопределяемые модули БСП.
2. **Проверка заполненности:** всегда исключать условно проверяемые реквизиты через УдалитьНепроверяемыеРеквизитыИзМассива; читать реквизиты через ЗначениеРеквизитаОбъекта/ЗначенияРеквизитовОбъектов, не через точку в цикле.
3. **Списки:** использовать УстановитьЭлементОтбораДинамическогоСписка для отборов; при отладке — получать текст запроса списка и ЗапросВСтрокуXML.
4. **Длительные операции:** не писать длительный код в модуле формы; размещать в модуле менеджера; использовать ПараметрыОжидания и ОповещениеПользователю для UX; при многопоточности обрабатывать соответствие результатов по ключам.
