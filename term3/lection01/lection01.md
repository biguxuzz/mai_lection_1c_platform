- Понимание управляемых форм и разделения клиент/сервер

### Блок 2: Зачем нужна БСП. Современная разработка на 1С (15 минут)

#### 2.1 Ошибка — игнорировать БСП (5 минут)

**Текущий уровень платформы:**
- Современная 1С — уровень enterprise: системы класса ERP, управление холдингом, тысячи пользователей
- Раньше: обычные формы, толстый клиент, возможность писать запросы в модуле формы
- Сейчас: управляемый интерфейс, жёсткое разделение клиент/сервер, асинхронность, отказ от модальных окон
- Сложность платформы выросла в разы; бороться с ней «вручную» без БСП — значит изобретать велосипед и проигрывать по скорости и качеству

**Позиция работодателей:**
- В вакансиях часто указано: «Отличное знание БСП»
- Бизнес считает деньги: незнание БСП ведёт к трате времени на то, что уже реализовано в фирме 1С
- Высокая зарплата и уважение коллег связаны с умением использовать типовой инструментарий

---

#### 2.2 Что такое БСП (5 минут)

**Библиотека стандартных подсистем** — набор универсальных функциональных подсистем. Они не являются законченным решением с предметной точки зрения: в одной только БСП нельзя «начать вести учёт». Это инструментарий для ваших решений.

**Тиражные решения (ERP, ЗУП, Бухгалтерия предприятия и др.):**
- Уже содержат встроенную БСП
- Разработчик дорабатывает и интегрирует объекты с подсистемами БСП

**Собственные решения:**
- Имеет смысл использовать БСП с самого начала
- Примеры: рассылка отчётов, гибкая настройка доступа (вплоть до записей в таблице), версионирование объектов — всё это есть в БСП
- Нет смысла дублировать функционал: код БСП качественный, протестированный, документированный

**Документация:**
- На its.1c.ru — инструкция по БСП с привязкой к версии; при разработке своего решения или работе с тиражным важно смотреть версию БСП и соответствующую инструкцию

---

#### 2.3 Метафора «торта»: архитектура решений на базе БСП (5 минут)

**Сверху вниз:**
- **Верх (украшения):** отраслевая специфика, доработки под конкретного заказчика
- **Средний слой (крем):** бизнес-логика — проводки, расчёты, документооборот
- **Основа (коржи):** БСП — база, фундамент

Менять можно «украшения» и «крем», но выдергивать нижний корж или подсовывать своё «сырое тесто» вместо использования БСП — значит рисковать устойчивостью системы. Новички часто «лепят свои коржики сбоку» вместо подключения к готовым подсистемам (файлы, уведомления, печать, права доступа и т.д.).

### Блок 3: Состав поставки БСП и варианты использования (15 минут)

#### 3.1 Что входит в поставку (5 минут)

**Файлы дистрибутива (после скачивания с ИТС):**
- **Демо (Demo CF/DT):** только для обучения и «пощупать» БСП на практике; развернуть демобазу, посмотреть примеры (объекты с префиксом «Демо»). В рабочей базе демо не используют.
- **1C V8 CF (конфигурация для российского рынка):** полная БСП с учётом региональной специфики; для массового внедрения и обновления в кастомных конфигурациях (не для получения «готовой рабочей базы» одной загрузкой).
- **International CF:** урезанный вариант (вырезано часть, относящаяся к РФ); меньше по размеру; для международных решений.
- **.ext / инструменты:** документация, инструкция по установке, обработки для внедрения и обновления БСП.

**Важно:** CF (кроме демо) не предназначены для «открыл пустую базу, загрузил CF — и база заработала». Внедрение и обновление выполняют по инструкции с сайта ИТС.

---

#### 3.2 Требования и развёртывание демонстрационной базы (5 минут)

**Платформа:**
- Версия технологической платформы не ниже указанной в описании выбранной редакции БСП (например, 8.5.1.1155)
- Скачивание платформы: Программы → 1С:Предприятие 8.3 → нужная сборка (например, для Windows)

**Развёртывание демо:**
1. 1С:Предприятие → Добавить → Создание новой информационной базы
2. Развернуть из шаблона → выбрать версию БСП (например, 3.2.1.319)
3. Указать имя и путь к базе → дождаться завершения
4. Открыть конфигурацию (Конфигурация → Открыть конфигурацию)
5. Первый запуск — под администратором (полные права)

---

#### 3.3 Первое внедрение БСП и обязательные подсистемы (5 минут)

**Обработка «Первое внедрение БСП»** (внешняя обработка из поставки):
- Показывает преднастройки: какие подсистемы можно включить при внедрении с нуля
- **Обязательные к внедрению (флаги не снимаются):**
  - Базовая функциональность
  - Обновление версии
  - Пользователи
- Остальные подсистемы — по зависимостям; в обработке отображаются зависимости «от кого / на кого» и необязательные связи
- Результат настройки — файл для сравнения/внедрения (само сравнение на лекции не выполняем; при самостоятельном внедрении с нуля — по инструкции ИТС)

**Внедрение БСП с нуля:** долгий процесс; большинство разработчиков работают с тиражными решениями, где БСП уже есть. Тем, кто внедряет с нуля, — строго по инструкции на ИТС.

### Блок 4: Самостоятельные и интегрируемые подсистемы (10 минут)

#### 4.1 Две категории подсистем

**Самостоятельные подсистемы:**
- Не требуют существенных доработок
- Предоставляют готовые командный и программный интерфейс

**Интегрируемые подсистемы:**
- Требуют внесения изменений в объекты, к которым подключаются
- Пример: «Запрет изменения по дате» — нужно доработать объект (форма, модуль, реквизиты) для тесной интеграции с подсистемой

---

#### 4.2 Приёмы интеграции объектов с БСП (10 минут)

**1. Вставки программного кода в модуле формы**  
В событиях (например, «При создании на сервере») вызываются методы программного интерфейса подсистем БСП (версионирование, контактная информация и т.д.). Других способов «подставить» свой код в модуль формы нет — поэтому используются прямые вызовы методов подсистем.

**2. Определяемые типы**  
Подсистема скрывает детали хранения (регистры, справочники). Объект подключается к подсистеме через определяемый тип (например, «Версионируемые данные», «Владелец присоединённых файлов»). В составе типа указывают свой документ/справочник; в форме добавляют немного кода и настройки — подсистема «оживляет» функционал (кнопки, хранение, выбор). Так же работают подписки на события через определяемые типы.

**3. Переопределяемые общие модули**  
Модули с постфиксом «переопределяемый» (например, «Заполнение объектов переопределяемый»). В них — описание объектов, интегрируемых в подсистему, и пустые процедуры. Вызывает эти процедуры БСП (принцип callback: «не вы звоните нам — мы вам перезвоним»). Задача разработчика — аккуратно вписать свой код в нужные процедуры.

**4. Интеграция через метаданные (контракт данных)**  
Подсистема задаёт жёсткие требования к структуре. Пример: подсистема «Контактная информация» — табличная часть с именем «КонтактнаяИнформация» и колонками строго по инструкции (тип, вид, значение). При соблюдении контракта подсистема сама добавляет валидацию, формы ввода и т.д.

### Блок 5: Правило фасада. Публичный и служебный код (15 минут)

#### 5.1 Публичная и служебная зоны

**Паттерн фасада:** весь код БСП разделён на две зоны.

- **Публичная (витрина):** модули и процедуры/функции без пометки «служебный»; это контракт, стабильный API. Только им и нужно пользоваться.
- **Служебная (внутренняя реализация):** в названии модуля или процедуры есть слово «служебный» либо модуль в подсистеме «Служебная». Это внутренности; вызывать их напрямую запрещено.

**Почему нельзя вызывать служебный код:** фирма 1С в любой момент может изменить внутреннюю реализацию без предупреждения. Гарантии и обратная совместимость — только для публичного интерфейса.

---

#### 5.2 Пример из практики

**Ситуация:** массовая печать документов на складе. Разработник нашёл в БСП процедуру в модуле «Управление печатью служебной», которая формировала массив табличных документов, и вызвал её напрямую. Задача сдана, полгода всё работало. После планового обновления БСП внутренняя процедура была переименована/разбита на две; вызов сломался — склад встал, ошибка «Метод не найден». Публичный интерфейс (модуль «Управление печатью» без «служебной») не менялся — при использовании его поломки бы не было.

**Вывод:** не обращаться к служебным модулям и служебным процедурам/функциям даже при наличии экспорта. В общих модулях обращать внимание на область «Служебные процедуры и функции» — её не использовать как API.

### Блок 6: Три способа легально «войти» в экосистему БСП (15 минут)

#### 6.1 Способ 1: Определяемые типы («магия типов»)

**Идея:** не вы зависите от модуля подсистемы, а подсистема готова принять любой объект, объявленный в составе определяемого типа.

**Пример (присоединённые файлы):**  
Нужно прикреплять к документу «Заявка на расход» файлы (сканы, договоры). Вместо своего справочника и своего кода: в метаданных находим определяемый тип «Владелец присоединённых файлов» → в составе типа отмечаем документ «Заявка на расход» → в модуль формы копируем несколько фрагментов кода и настраиваем типы данных по инструкции. В форме появляется кнопка, выбор файлов, drag-and-drop; логику хранения реализует БСП.

#### 6.2 Способ 2: Переопределяемые модули (callback)

БСП «спрашивает совета» у конфигурации: при старте системы, при проверке дублей партнёра и т.д. В переопределяемых модулях — пустые процедуры; разработчик вписывает свой код. Вызов выполняет только БСП, сам разработчик эти процедуры не вызывает.

#### 6.3 Способ 3: Интеграция через метаданные (контракт данных)

Жёсткий вариант: подсистема задаёт структуру (имя табличной части, имена и типы колонок). Пример — «Контактная информация»: табличная часть с заданными колонками (тип, вид, значение). Ошибка в названии или типе — функционал не заработает; при точном соблюдении контракта подсистема сама подключает формы и валидацию.

### Блок 7: Собственные объекты для практики (15 минут)

#### 7.1 Включение изменений и своя подсистема

Чтобы добавлять свои объекты в демонстрационную конфигурацию БСП:
- Конфигурация → Поддержка → Настройка поддержки → включить возможность изменения конфигурации
- В настройках правил поддержки для корня конфигурации: «Не редактируется» → снять, включить редактирование

Добавляем подсистему, например «Примеры подключения», включаем её в командный интерфейс.

---

#### 7.2 Пример: справочники и документы

**Справочники:**
- «Склады» (длина кода 3, наименование 150) — включить в подсистему «Примеры подключения»
- «Складские помещения» — владелец «Склады», включить в ту же подсистему

**Документы:**
- «Приобретение товаров и услуг»: шапка — реквизит «Склад» (СправочникСсылка.Склады); табличная часть «Товары» — «Складское помещение» (связь по владельцу со складом из шапки), «Номенклатура» (demo_nomenclatura), «Количество» (15.3, не отрицательное)
- «Реализация товаров и услуг»: склад перенести в табличную часть (как развитие существующей системы), связь параметров выбора складского помещения — с табличной частью

Так мы получаем базу для последующих занятий по подключению к подсистемам БСП (версионирование, файлы, печать и т.д.).

### Блок 8: БСП как учебник и практические приёмы (10 минут)

#### 8.1 Чтение кода БСП

БСП можно рассматривать как лучший «учебник» по программированию на 1С: код писали методологи платформы, в нём отражены десятилетия опыта. Рекомендация: 15–20 минут в день читать код (например, общий модуль «Общего назначения») — не искать сразу решение задачи, а изучать стиль, именование, обработку ошибок, использование структур и соответствий. Со временем начнут узнаваться паттерны (стратегия, адаптер, фабрика и т.д.) и улучшится собственный код.

#### 8.2 Полезные функции БСП (примеры)

- **Значения реквизитов объектов** — чтение реквизитов через запрос, без выгрузки всего объекта (оптимизация вместо «объект.Реквизит»).
- **Установить отбор динамического списка** — корректная установка отбора без дублирования и конфликтов правых значений.
- **Интервал времени строкой** — человекопонятное представление разницы между датой начала и окончания (например, «больше года»).
- **Подсистема «Свойства» (дополнительные реквизиты):** добавление полей (цвет, размер, бренд и т.д.) без изменения метаданных и перехода в конфигуратор — через планы видов характеристик; форма перестраивается «на лету», база не блокируется. Понимание этого механизма помогает переходить от «хардкода» к настройкам.

Перед написанием своего кода полезно спросить: «Какую подсистему БСП можно применить к этой задаче?» (в т.ч. с помощью поиска и ИИ в поисковиках). При добавлении нового документа/справочника — смотреть в типовой конфигурации, как подключены стандартные команды (печатные формы, сервисные кнопки и т.д.).

### Блок 9: Резюме и боевые задачи (10 минут)

#### 9.1 Краткое повторение

- БСП — фундамент современных решений 1С; игнорировать её в 2026 году — профессиональная ошибка.
- БСП — не готовая учётная система, а набор подсистем (файлы, уведомления, доступ, версионирование, печать, свойства и др.); их используют и в тиражных, и в собственных конфигурациях.
- Внедрение: по инструкции ИТС; демо — только для обучения; служебный код не вызывать, использовать только публичный API (правило фасада).
- Интеграция: определяемые типы, переопределяемые модули, контракт данных через метаданные.
- БСП полезна и как «учебник»: регулярное чтение кода повышает культуру разработки.

#### 9.2 Практические шаги

## Резюме лекции (5 минут)

### Ключевые моменты

1. **БСП** — набор универсальных подсистем, а не готовая конфигурация; обязательна для современной разработки и востребована работодателями.
2. **Поставка:** демо (только обучение), 1C V8 CF (РФ), International CF; внедрение и обновление — по инструкции ИТС.
3. **Обязательные при внедрении с нуля:** базовая функциональность, обновление версии, пользователи; остальное — по зависимостям.
4. **Подсистемы:** самостоятельные (готовый интерфейс) и интегрируемые (требуют доработки объектов).
5. **Приёмы интеграции:** вставки кода в модуль формы, определяемые типы, переопределяемые общие модули, контракт данных (метаданные).
6. **Правило фасада:** использовать только публичный API; служебный код не вызывать.
7. **Три легальных способа «войти» в БСП:** определяемые типы, переопределяемые модули, интеграция через метаданные.
8. **Практика:** собственная подсистема и объекты (справочники, документы) в демобазе для последующих занятий; чтение кода БСП как регулярная привычка.
