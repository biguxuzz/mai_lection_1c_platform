### Блок 2: Обзор платформы 1С:Предприятие (15 минут)

#### 2.1 Что такое 1С:Предприятие (5 минут)

**1С — платформа бизнес-ориентированной разработки:**
- Позиционируется как low-code, но в основном все-таки приходится кодировать
- Предметно-ориентированное программирование (Domain-Specific Language)
- Готовый набор объектов (справочники, документы, регистры), из которых "как из кубиков" выстраивается система

**Типы программирования:**
1. Системное (на уровне ОС, ассемблер)
2. Функциональное (функции, процедуры, простой код)
3. Объектно-ориентированное (создание объектов, наследование)
4. **Предметно-ориентированное** — уже придуманные объекты, используется как конструктор

**Преимущества:**
- Достаточно быстро создать простой продукт
- Легко поддерживать
- Быстрый старт благодаря готовому набору объектов

---

#### 2.2 История развития платформы (10 минут)

**90-е годы:**
- Создание фирмы 1С
- Первые версии — закрытые коробочные решения
- Версия 6: возможность подправлять формы в пользовательском режиме (конфигуратор еще не отделялся)
- Сначала для MS-DOS, затем для Windows

**Предприятие 7 (1998):**
- Клиент-серверный вариант
- Толстый клиент для работы
- Разделение данных:
  - Конфигурация — отдельный файл
  - Данные — DBF или подключение к MS SQL
- Крупные решения размещались в базе MS SQL
- Типовые продукты: Бухгалтерия, Торговля и Склад, Зарплата и Кадры
- Решения от сети франчайзи: "Бухгалтерия + Услуги + Предприятие"

**Платформа 8 (2002):**
- Трехзвенная архитектура:
  - Сервер СУБД (MS SQL, Oracle, DB2)
  - Application сервер (бэкенд-часть)
  - Frontend (тонкий клиент или браузер)
- Переименование типовых конфигураций:
  - Бухгалтерия предприятия
  - Управление торговлей
  - Управление производственным предприятием
  - Зарплата и управление персоналом

**ERP (самый распространенный продукт):**
- Монолит, включающий:
  - Расчет зарплаты
  - Бухгалтерский учет
  - Формирование отчетности для государства
  - Автоматизация планирования производства
  - Посменное планирование
  - Наладка цеховой работы
  - Контроль потребления материалов

### Блок 4: Преимущества разработки на платформе 1С (10 минут)

#### 4.1 Востребованность и карьера

**Текущая ситуация:**
- Массовый переход с Oracle, SAP на отечественные рельсы
- Огромное наследство больших КС требует перевода
- Очень большой спрос на специалистов
- Без работы 1С-нику сложно остаться

**Карьерные пути:**
- Старт карьеры в качестве поддержки конечного пользователя на предприятии
- Консалтинг (более сложный вход)
- Чем выше квалификация — тем выше зарплата и спрос

#### 4.2 Открытость и сообщество

**Открытость:**
- Практически вся документация открыта
- Код конфигурации открыт (если не решения от франчайзи с закрытыми модулями)
- Комьюнити-лицензия доступна открыто

**Сообщество:**
- Infostart (аналог Stack Overflow в сфере 1С)
- Огромное количество разработок (бесплатно и платно)
- Сторонние разработчики создают интеграции (маркетплейсы, маркировка)

**Комьюнити-лицензия:**
- Доступна серверная версия 1С
- Подключение возможно с вашего компьютера (без VS) или с графической частью
- Нет ограничений по размеру базы
- PostgreSQL бесплатная версия для 1С доступна

#### 4.3 Язык и скорость разработки

**Русский язык:**
- Все на русском (хотя можно разрабатывать и на английском)
- ERP International Edition — весь код на английском
- Библиотека стандартных подсистем выпускается на английском

**Быстрый старт:**
- Готовый набор объектов
- Не нужно обучать пользователей долго (достаточно быстро начинают использовать)

**Терминология:**
- Много специфичных терминов (ITIL, CRM, ЭДО, ERP и т.д.)
- Специфичный "птичий язык" специалистов

#### 4.4 Предметная область

**Важно:**
- Ценится не только умение разрабатывать на платформе
- Знание предметной области, которую автоматизируете
- Специфичные области: медицина, строительство, производство (MES-системы), ремонты оборудования (ТОИР)

**Универсальные решения:**
- ERP + УХ (все в одном комплекте)
- Бюджетирование от Управления холдингом
- Планирование ресурсов
- Расчеты с персоналом (обычно выносятся в отдельную конфигурацию)

### Блок 6: Состав информационной базы (10 минут)

#### 6.1 Конфигурации в информационной базе

**Типы конфигураций:**

1. **Конфигурация информационной базы:**
   - С чем работают текущие пользователи

2. **Конфигурация разработчика:**
   - Пока пользователи работают, разработчик может подключиться и конфигурировать
   - После применения конфигурация разработчика накатывается на информационную базу
   - Две разные конфигурации содержатся в информационной базе

3. **Конфигурация поставщика:**
   - Если используется типовой продукт (например, Управление нашей фирмой)
   - Нужна для сравнения при обновлениях
   - Позволяет увидеть, что было добавлено или изменено разработчиком

**Выгрузка конфигураций:**
- Можно выгрузить отдельно (получите CF файл — файл конфигурации)
- Пользовательские данные отдельно выгружаются сложнее (используется библиотека стандартных подсистем)

**Важно для только пользовательских данных:**
- При загрузке данных в другую область конфигурация должна быть идентична
- Нельзя загрузить данные из версии 3.0.96 в версию 3.0.100
- Нужно сначала догнать конфигурацию до нужной версии

#### 6.2 Варианты развертывания

**DT — выгрузка информационной базы:**
- Вместе с данными, пользователями и всеми конфигурациями
- Можно развернуть на другой СУБД

**Файловая информационная база:**
- Не требует отдельной инсталляции СУБД
- Подходит для одновременной работы до 10 пользователей
- Особенно если пользователи работают с разными областями (бухгалтер, менеджер, расчетчик зарплаты, кадровик)
- После 10 пользователей работать становится сложно, особенно при высокой нагрузке

### Блок 7: Клиент-серверная архитектура (15 минут)

#### 7.1 Компоненты клиент-серверной развертки

**СУБД:**
- Отдельно выделяется СУБД
- Может быть отказоустойчивой (при выходе одного сервера переключение на другой)

**Кластер серверов 1С:**
- При установке серверной части на одной машине поднимается кластер
- Несколько служб:
  - Одна отвечает за подключение пользователей
  - Другая — за коннект с СУБД
  - Третья — за лицензии
- Горизонтально масштабируем:
  - Распределение нагрузки
  - Полное дублирование (синхронизация сеансовых данных)
  - При отключении одного сервера пользователь не заметит

**Веб-сервис:**
- Практически все крупные внедрения используют веб-сервис
- Публикация информационной базы
- Пользователи могут работать через браузер

#### 7.2 Мультиплатформенность

**Операционные системы:**
- MS SQL Server — в основном на Windows (на Linux есть инсталляция, на BSD вряд ли)
- PostgreSQL — отлично работает на Linux
- PostgreSQL на Windows — звучит странно, но встречается
- Крупные клиенты активно используют Astra Linux

**Веб-сервис:**
- Публикуется через Apache
- Балансировщик можно сделать через Nginx
- Веб-сервер выбирает доступный или незагруженный сервер
- Кластер серверов 1С выбирает СУБД
- Кластер серверов 1С отлично работает на Linux
- На Mac — только файловая база (сервер пока не получалось развернуть)

**Архитектура балансировки:**
```
Пользователь → Балансировщик → Веб-сервис → Кластер серверов 1С → СУБД
```

### Блок 10: Роли на проекте (10 минут)

#### 10.1 Функциональные роли

**Консультант/Аналитик:**
- В консалтинге: консультант выше по грейду, чем аналитик
- В 1С-сфере (например, Рарус, БИТ): консультант знает определенную область 1С, аналитик описывает область и выбирает архитектуру

**Технический писатель:**
- Редко встречается как отдельная роль
- Обычно аналитик/консультант выступает техническим писателем

**Функциональный архитектор:**
- Не программирует, но понимает, какую архитектуру выбрать
- На крупных проектах — по определенной области (например, функциональный архитектор области производства или закупок)
- Может оценить стоимость доработок и внедрения по времени

#### 10.2 Технические роли

**Администратор 1С (базисник в терминах SAP):**
- Умеет развернуть всю структуру
- Настроить отказоустойчивость серверов СУБД и кластера 1С
- Подключить к веб-серверу
- Настроить балансировщики
- Не очень понимает функциональность и не лезет туда

**Тестировщик:**
- В консалтинге обычно консультант/аналитик

**Разработчик:**
- Действует по ТЗ
- Взаимодействует с консультантом более высокого грейда

**Ведущий разработчик:**
- Может принимать решения, как разрабатывать
- Сам разрабатывает

**Технический архитектор:**
- Эксперт по технологическим вопросам
- Оценка: какие серверы потребуются, мощность, процессоры, планирование потребления
- Дисковая система
- Взаимодействие на низком уровне с СУБД
- Поиск мест замедления работы системы
- Оптимизация

#### 10.3 Управление проектом
